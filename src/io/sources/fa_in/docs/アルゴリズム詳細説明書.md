# fa_in アルゴリズム詳細説明書

## Source Selection

1. ROS2 node が `backend.name` から source backend を明示生成する。
2. `alsa_capture` backend の場合、ALSA の PCM capture source を列挙する。
3. `pcm_file_reader` backend の場合、`file.path` と `audio.source_id` を必須として raw PCM file source を構成する。
4. `network_pcm_receiver` backend の場合、`endpoint.uri`、`transport.identity`、`audio.source_id` を必須として raw PCM UDP source を構成する。
5. `audio.device_selector.mode` が `id` の場合は raw ALSA id の完全一致で source を決める。`name` の場合だけ表示名の完全一致も許可する。
6. `index` の場合は列挙結果の index で source を決める。
7. source が決まらない場合は起動を失敗させる。

`auto` source selection は使いません。入力源が変わると後段のデータ意味が変わるためです。

ALSA source id は backend open 境界でも `hw:` で始まる raw hardware id に限定する。列挙結果を経由せず直接 open された場合でも、`default`、`plug*`、`plughw*`、`pulse`、`pipewire` などの plugin PCM は fail closed にする。

## Device Switch Selection

`switch_device` service は `target_selector_mode` によって `id` / `name` / `index` を明示させる。`id` mode は raw ALSA id の完全一致だけを扱い、表示名探索へ fallback しない。`name` mode の場合だけ表示名の完全一致を使う。`index` mode は `target_index` だけを使う。mode と field の組が不正な場合、または選択 source が解決できない場合は失敗応答を返し、別 selector の探索へ fallback しない。

選択 source が解決できた場合、service は必ず capture thread を止め、backend を reopen し、成功時だけ `active_device_id` を更新して capture thread を再開する。reopen せずに「選択だけ成功」とする no-op mode は持たない。

## Device Listing

`list_devices` service は backend の `DeviceInfo` に含まれる id、表示名、capability をそのまま response に写す。`max_input_channels` / `default_sample_rates` は backend が列挙時に取得できた値だけを返す。

ALSA hint enumeration だけでは capability を安全に確定できないため、ALSA backend の capability は unknown として `0` になる。`fa_in` node は起動中の `audio.channels` / `audio.sample_rate` を device capability として合成しない。設定済み stream format は `AudioFrame` metadata で表す。

`pcm_file_reader` backend は device enumeration を持たない。`list_devices` は開いている file source の identity だけを返し、device selector による切替は許可しない。

`network_pcm_receiver` backend も physical device enumeration を持たない。`list_devices` は開いている endpoint identity だけを返し、`switch_device` は許可しない。network endpoint の切替は source の意味を変えるため、別 config として起動し直す。

## Device Configuration

1. ROS2 node は source id、format、requested period frames を backend に渡す。
2. `alsa_capture` backend は指定 ALSA id を `SND_PCM_STREAM_CAPTURE` で open する。
3. access は interleaved PCM に固定する。
4. encoding / bit depth pair から ALSA format を決定する。
5. channel count / sample rate / period size を設定する。
6. requested period frames が ALSA frame count 型の範囲を超える場合は、ALSA 呼び出し前に起動を失敗させる。
7. requested period frames が ALSA negotiation で変わった場合は起動を失敗させる。
8. 設定に失敗した場合は起動を失敗させる。

`pcm_file_reader` backend は configured `file.path` を binary mode で開き、file size が expected frame byte size で割り切れることを確認する。file payload の decode、resample、channel conversion、gain は行わない。

`network_pcm_receiver` backend は configured `endpoint.uri` を `udp://IPv4:port` として parse し、UDP socket を bind する。受信 packet は expected frame byte size で割り切れる場合だけ publish 対象にする。packet 未着は `polling.period_ms` 待機で同一 endpoint を再 poll し、空 packet、oversized packet、partial frame packet は fail closed にする。jitter buffer、PLC、clock drift correction は行わない。

`audio.chunk_ms` は `audio.sample_rate * audio.chunk_ms / 1000` が整数 frame count になる場合だけ受け付ける。node 内で最小 frame 数への丸めや切り捨ては行わない。

bytes per frame と capture buffer byte count は、buffer allocation 前に表現可能範囲を検証する。`audio.channels * audio.bit_depth / 8` や `frames_per_buffer * bytes_per_frame` の overflow は起動失敗として扱う。

## Frame Publishing

1. capture thread が source backend から chunk を読む。
2. XRUN / negative error / zero-frame read は required input source の破断として扱い、fatal 状態へ移行する。
3. ALSA backend では read frames が configured capture chunk と完全一致することを確認し、byte count を計算する。file backend では EOF 前の短い final chunk を許可し、network backend では UDP packet size に応じた可変 frame count を許可し、read frames から byte count を計算する。
4. `AudioFrame.header.stamp`, `source_id`, `stream_id`, `encoding`, `sample_rate`, `channels`, `bit_depth`, `layout`, `data` を設定する。
5. `output_topic` へ publish する。

`output_topic` は ROS2 transport topic、`audio.stream_id` は `AudioFrame.stream_id` として扱う stream identity である。両者が同一 identity になる設定は起動時に fail closed する。

`fa_in` は波形値の解析を行いません。RMS / peak / VAD は専用 processing node で扱います。

ALSA source では短い read を可変 chunk として publish しません。`audio.chunk_ms` で定義された frame duration が後段の buffer / VAD / ASR 契約になるため、configured capture chunk と異なる read は required live source の runtime failure として fail closed します。file source の EOF 前 final chunk は source 終端を表すため、別 source へ fallback せず publish 後に EOF を待ちます。network source は packet boundary を入力境界として扱うため、`network.max_packet_bytes` 以下の可変 packet を 1 `AudioFrame` として publish します。

## Runtime Failure Handling

runtime read failure では `snd_pcm_prepare` による自動復旧や別 source への reopen を行いません。`fa_in` は adapter であり、source path の破断を後段に隠すと timestamp と入力意味が壊れるため、fatal log を出して ROS shutdown を要求します。

## Transport QoS

AudioFrame publisher は `audio.qos.depth` / `audio.qos.reliable` を必須設定として読み、`SensorDataQoS` や固定 reliability へ fallback しない。
diagnostics publisher は `diagnostics.qos.depth` / `diagnostics.qos.reliable` を必須設定として読み、system default QoS へ fallback しない。

## Backend Boundary

`include/fa_in/backends/source_backend.hpp` は ROS-free な source backend interface です。`fa_in_node` は backend を呼び出して device 管理と byte stream read を行い、ROS message への変換だけを担当します。

`src/backends/alsa_capture_backend.cpp` は ALSA API を直接呼ぶ唯一の実装です。このファイルは `rclcpp`、`fa_interfaces`、ROS message header を include しません。

`src/backends/pcm_file_reader_backend.cpp` は filesystem / ifstream だけを扱う ROS-free backend です。`file.path`、file size、byte-aligned frame contract、EOF / loop を扱い、ROS topic や `AudioFrame` への変換は node 側だけが担当します。

`src/backends/network_pcm_receiver_backend.cpp` は POSIX socket だけを扱う ROS-free backend です。`endpoint.uri` parse、socket open / bind、non-blocking receive、byte-aligned frame contract を扱い、ROS topic や `AudioFrame` への変換は node 側だけが担当します。
