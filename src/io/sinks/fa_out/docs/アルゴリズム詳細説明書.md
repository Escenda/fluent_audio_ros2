# fa_out アルゴリズム詳細説明書

## Device Configuration

1. ROS2 node が parameter を検証し、`backend.name` ごとの typed backend config を `SinkBackendSettings` として作る。
2. `alsa_playback` の場合、ROS-free backend が `audio.device_id` で ALSA playback device を open する。
3. node と backend が `audio.encoding=PCM16LE` と `audio.bit_depth=16` を検証する。
4. backend が ALSA software resampling を無効にする。
5. backend が interleaved access, `SND_PCM_FORMAT_S16_LE`, channels, sample rate を設定する。
6. backend が `audio.alsa.buffer_frames` と `audio.alsa.period_frames` を設定する。
7. ALSA negotiation が buffer / period frames を要求値から変更した場合は起動を失敗させる。
8. backend が software params を設定する。warning 継続は行わず、失敗時は起動を失敗させる。
9. backend open result は info message のみを返す。warning で継続する経路は持たない。
10. `pcm_file_writer` の場合、ROS-free backend が `file.path` を raw PCM output target として open する。
11. target file の親 directory がない、target が directory、または `overwrite.enabled=false` で既存 file がある場合は起動を失敗させる。
12. `network_pcm_sender` の場合、ROS-free backend が `endpoint.uri` を `udp://IPv4:port` として parse し、UDP send socket を open する。
13. `network.max_packet_bytes` が expected frame byte size で割り切れない場合は起動を失敗させる。
14. 設定できない場合は起動を失敗させる。

`audio.sample_rate * audio.chunk_duration_ms` は整数 playback frame count になる必要があります。`fa_out` は backend write chunk を隠れて丸めたり、1 sample などへ下限補正しません。

bytes per frame と playback chunk byte count は、playback thread 起動前に表現可能範囲を検証します。`audio.channels * audio.bit_depth / 8` や `playback_chunk_frames * bytes_per_frame` の overflow は起動失敗として扱います。

## Frame Validation

1. `stream_id` が `input_stream_id` と一致することを確認する。
2. encoding が backend config と一致することを確認する。
3. sample rate / channels / bit depth が node config と一致することを確認する。
4. data が空ではないことを確認する。
5. data byte count が frame size で割り切れることを確認する。

一致しない frame は playback queue に入れません。

## Playback Queue

1. valid frame を queue に入れる。
2. queue が上限を超える場合、frame drop で継続せず fatal 状態へ移行する。
3. playback thread が queue から frame を取り出し、speaker / file backend では chunk 単位で backend の write API に渡す。
4. network backend では accepted `AudioFrame` 1 件を 1 UDP packet として backend の write API に渡す。
5. `snd_pcm_writei` が XRUN / EAGAIN / negative error / zero-frame write を返した場合、required sink の破断として fatal 状態へ移行する。
6. frame 全体を write できた場合、元 `AudioFrame.header`、`header.frame_id` 由来の request_id、`AudioFrame.epoch` を `PlaybackDone` として publish する。

playback thread は runtime 中に device handle が失われても自動 reopen しません。`fa_out` は sink adapter であり、別 device や ALSA plugin 経由の復旧で出力先の意味を変えないためです。

ALSA、file writer、network sender の concrete 生成は `include/fa_out/backends/factory.hpp` と `src/backends/factory.cpp` に閉じ込めます。`backend.name` の missing / unknown はここでも fail closed にし、node は `SinkBackendSettings` と `SinkBackend` のみを知ります。ALSA backend だけは node contract test のために factory injection を受け付けますが、file / network backend の concrete class は node から直接生成しません。

ALSA、file writer、network sender の open / write / close は `include/fa_out/backends` と `src/backends` の backend class に閉じ込めます。backend は ROS2 topic、ROS message、`rclcpp` を include しません。

backend handle への write / isRunning / close は node 側の `backend_mutex_` で直列化します。queue overflow など ROS callback thread 側で fatal 状態へ移行する経路でも、playback thread が write 中の backend を同時に close / reset しないためです。

## Playback Control

`PlaybackControl` service は `stop`、`pause`、`resume` のみを受け付けます。

- `stop`: `active_epoch_` を1つ進め、queue を空にし、pause を解除する。playback thread が旧 epoch frame を write 中の場合、次の chunk boundary で stale として中断する。
- `pause`: backend への新規 chunk write を止める。queue の frame は破棄しない。
- `resume`: pause を解除し、queue に残っている active epoch 以上の frame の write を再開する。

未知 command は `accepted=false` とし、音声出力の状態は変更しません。epoch が `uint32` 上限に達して `stop` できない場合、旧 frame と新 frame の境界を保証できないため fail closed します。

## Sink Adapter Boundary

`fa_out` が見るのは `input_topic` と `input_stream_id` で明示された frame contract と、playback lifecycle に必要な `AudioFrame.header.frame_id` / `AudioFrame.epoch` だけです。`input_topic` は ROS transport の接続点、`input_stream_id` は frame identity であり、同一値または先頭 `/` の有無だけが違う同一 identity は設定エラーとして起動失敗にします。barge-in や複数 stream の routing はここで解釈しません。

`fa_out` の lifecycle 制御は queue / chunk boundary に限定します。network sender では packet boundary が `AudioFrame` 単位になるため、speaker / file backend のような chunk 分割は行いません。stop / pause / resume のために decode、resample、gain、routing、TTS、別 sink への fallback は行いません。ALSA backend、PCM file writer backend、network PCM sender backend は ROS-free の sink backend として残し、playback lifecycle の ROS message / service は node adapter が扱います。
