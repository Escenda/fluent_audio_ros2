# fa_out アルゴリズム詳細説明書

## Device Configuration

1. ROS2 node が parameter を検証し、`alsa_playback` backend config を作る。
2. ROS-free backend が `audio.device_id` で ALSA playback device を open する。
3. node と backend が `audio.encoding=PCM16LE` と `audio.bit_depth=16` を検証する。
4. backend が ALSA software resampling を無効にする。
5. backend が interleaved access, `SND_PCM_FORMAT_S16_LE`, channels, sample rate を設定する。
6. backend が `audio.alsa.buffer_frames` と `audio.alsa.period_frames` を設定する。
7. ALSA negotiation が buffer / period frames を要求値から変更した場合は起動を失敗させる。
8. backend が software params を設定する。warning 継続は行わず、失敗時は起動を失敗させる。
9. backend open result は info message のみを返す。warning で継続する経路は持たない。
10. 設定できない場合は起動を失敗させる。

`audio.sample_rate * audio.chunk_duration_ms` は整数 playback frame count になる必要があります。`fa_out` は backend write chunk を隠れて丸めたり、1 sample などへ下限補正しません。

bytes per frame と playback chunk byte count は、playback thread 起動前に表現可能範囲を検証します。`audio.channels * audio.bit_depth / 8` や `playback_chunk_frames * bytes_per_frame` の overflow は起動失敗として扱います。

## Frame Validation

1. `stream_id` が `audio/output/frame` と一致することを確認する。
2. encoding が `PCM16LE` と一致することを確認する。
3. sample rate / channels / bit depth が node config と一致することを確認する。
4. data が空ではないことを確認する。
5. data byte count が frame size で割り切れることを確認する。

一致しない frame は playback queue に入れません。

## Playback Queue

1. valid frame を queue に入れる。
2. queue が上限を超える場合、frame drop で継続せず fatal 状態へ移行する。
3. playback thread が queue から frame を取り出し、chunk 単位で backend の write API に渡す。
4. `snd_pcm_writei` が XRUN / EAGAIN / negative error / zero-frame write を返した場合、required sink の破断として fatal 状態へ移行する。

playback thread は runtime 中に device handle が失われても自動 reopen しません。`fa_out` は sink adapter であり、別 device や ALSA plugin 経由の復旧で出力先の意味を変えないためです。

ALSA の open / write / close は `include/fa_out/backends` と `src/backends` の backend class に閉じ込めます。backend は ROS2 topic、ROS message、`rclcpp` を include しません。

backend handle への write / isRunning / close は node 側の `backend_mutex_` で直列化します。queue overflow など ROS callback thread 側で fatal 状態へ移行する経路でも、playback thread が write 中の backend を同時に close / reset しないためです。

## Sink Adapter Boundary

`fa_out` は stop / pause / resume topic、playback_done / paused notification、stale timestamp / epoch filtering を持ちません。これらは session controller、routing node、または upstream queue controller の責務です。

`fa_out` が見るのは `audio/output/frame` の frame contract だけです。frame の順序、barge-in、completion notification、request epoch の意味はここで解釈しません。
