# fa_jitter_buffer アルゴリズム詳細説明書

## 入力検証

処理は `handleFrame()` で始まる。null pointer、必須 field 欠落、topic identity mismatch、format mismatch、byte alignment mismatch、FLOAT32LE sample 異常はいずれも drop とし、buffer へ入れない。

この node は無効 frame を補正しない。欠落した field を default で埋める、空 data を silence とみなす、format mismatch を変換する、stale frame を現在 frame として扱うことはしない。

## active stream

active stream は次の値で定義する。

- `source_id`
- `stream_id`
- `sample_rate`
- `channels`
- `encoding`
- `bit_depth`
- `layout`

valid frame が active stream と異なる場合、現在の buffer を reset してから新しい stream として扱う。異なる source / format の frame を同じ queue へ混ぜないためである。

## epoch queue

buffer は `std::map<uint32_t, AudioFrame>` で保持する。key は `AudioFrame.epoch` であり、publish は常に `begin()` の最小 epoch から行う。

受信順が `10, 12, 11` の場合でも、target depth を超えるまで保持されていれば publish 順は `10, 11, 12` になる。

## publish condition

valid frame を挿入した後、次の条件を満たす限り publish する。

```text
buffered_frames > jitter.target_depth_frames
```

publish 時は frame を copy し、`stream_id` だけを `output_topic` に更新する。音声 payload、timestamp、source、format、epoch は変更しない。

## duplicate / late frame

duplicate は次のどちらかである。

- 同じ epoch が buffer 内に存在する。
- 同じ epoch がすでに publish 済みである。

late frame は publish 済み epoch より古い epoch を持つ frame である。通常は drop する。`jitter.reset_on_epoch_regression=true` の場合だけ、late frame を stream epoch reset として扱い、buffer と publish 済み epoch を reset してから受け入れる。

## max depth

`jitter.max_depth_frames` は設定契約の上限であり、`target_depth_frames` より大きい必要がある。この node は target depth を超えた分を即座に publish するため、通常運用で queue は target depth 以内に戻る。

## diagnostics

diagnostics は設定値、現在の `buffered_frames`、および counter を publish する。

- `frames_in`
- `frames_out`
- `frames_dropped`
- `duplicate_drops`
- `late_drops`
- `resets`
