# fa_audio_window テスト設計

## 0. この資料の位置づけ

この資料は `fa_audio_window` の現在実装済み slice に対して、何を検証するべきか、現在どこまで検証済みかを整理する。

テスト設計は test code を貼る場所ではない。ここでは、検証対象の性質、前提、期待結果、失敗条件、未検証範囲を記述する。

## 1. 検証対象

`fa_audio_window` で守るべき主な性質は次である。

- PCM16LE `AudioFrame` を rolling window として保持できること。
- retention 外の frame を保持対象から外すこと。
- numeric time range から sample boundary に沿って PCM data を抽出できること。
- empty / outside / non-continuous window を成功扱いしないこと。
- export / archive service が明示 error contract を返すこと。
- unsupported scope / unsupported format / missing archive reason を fail closed にすること。
- WAV output が PCM16LE/WAV contract を満たすこと。
- export / archive identity が決定的であること。
- 同一 request の再実行で同じ `AudioClipRef` を返し、sequence-based refs を増やさないこと。
- 既存 deterministic target が同一 bytes / metadata なら成功し、異なる場合は明示失敗すること。
- `AudioClipRef.content_sha256` が WAV bytes の SHA-256 であること。
- archive の `AudioClipRef.metadata_sha256` が publish 済み metadata JSON bytes の SHA-256 であること。
- metadata JSON 内の `audio_clip_ref` が自己参照になる hash fields を含まないこと。
- `filesystem` archive store が store directory へ deterministic / no-clobber publish すること。
- `filesystem` archive store の設定不備と `file://` URI prefix を起動時 fail closed にすること。

## 2. 現在の test binary

現在の package test は 3 つの C++ test binary で構成される。

| test binary | 主な検証 |
| --- | --- |
| `test_audio_window_buffer` | retention、timestamp order、range query、empty window、sample boundary、non-continuous gap、overlap rejection |
| `test_wav_writer` | PCM16LE WAV header と payload |
| `test_export_service_contract` | ROS service contract、explicit failure、deterministic export/archive identity、filesystem archive store、conflict failure |

`test_export_service_contract` は 11 tests を持つ。

## 3. Buffer / range query test

目的:

- retained window が requested range を正しく coverage できることを証明する。
- coverage できない状態を空 clip 成功にしないことを証明する。

検証観点:

| 観点 | 期待 |
| --- | --- |
| retention | retention 外 frame は query 対象から外れる |
| timestamp order | timestamp 順に frame を保持し、requested range の sample を順序通り返す |
| empty window | `window_not_found` 相当の状態になる |
| non-aligned start/end | sample boundary に合わせて contained samples を返す |
| non-continuous gap | gap を持つ range は non-continuous failure |
| overlapping frame | overlap frame を拒否し、retention state を進めない |

## 4. WAV writer test

目的:

- PCM16LE samples を RIFF/WAVE として保存する際の header と data chunk が contract を満たすことを証明する。

検証観点:

- `RIFF` / `WAVE` / `fmt ` / `data` chunk が期待通りであること。
- sample rate、channel count、bit depth、byte rate、block align が input format と一致すること。
- PCM payload bytes が変化しないこと。
- unsupported format や aligned でない PCM data は成功扱いにしないこと。

## 5. Service contract test

目的:

- ROS service surface が package contract 通りに成功 / 失敗を返すことを証明する。
- 決定的 identity と conflict policy を実行経路で確認する。

現在の検証観点:

| 観点 | 期待 |
| --- | --- |
| explicit failure contracts | parse failure、empty window、outside window、non-continuous window などが明示 error になる |
| repeated export | 同一 export request が同じ `AudioClipRef` を返し、WAV file は増えない |
| export conflict | 既存 deterministic WAV bytes が異なると `export_failed` |
| duplicate service names | export / archive service name 同一設定は起動時拒否 |
| repeated archive | 同一 archive request が同じ `AudioClipRef`、同じ WAV bytes、同じ metadata を返す |
| missing archive metadata | 既存 WAV bytes が同一で metadata が欠損している場合、metadata を no-clobber publish で再作成する |
| archive conflict | 既存 WAV bytes または metadata が異なると `archive_failed` |
| archive media identity hashes | `content_sha256` / `metadata_sha256` が publish 済み bytes と一致する |
| metadata self-reference avoidance | metadata JSON 内の `audio_clip_ref` は `metadata_uri` / hash fields を含まない |
| filesystem store publish | `filesystem` backend が store directory へ WAV / metadata を no-clobber publish し、非 `file://` URI prefix を返す |
| filesystem store conflict | store directory の既存 WAV / metadata が差分を持つと `archive_failed` |
| filesystem store config failure | backend unknown、必須設定欠損、`file://` prefix は起動時に拒否する |
| archive explicit errors | missing reason、unsupported scope / format、window failure が明示 error になる |

## 6. 決定的 identity regression

この package では sequence 採番による clip ref 増殖を禁止する。regression test では次を守る。

- 同じ operation、window id、window epoch、source id、stream id、resolved scope、resolved exported range、codec、container、payload format から同じ `clip_id` / path が生成される。
- archive では reason と related artifact ids が identity に含まれる。
- 同じ archive request の再実行は metadata を再解釈して同一性を確認する。
- conflict は上書きではなく fail closed とする。
- `content_sha256` は export / archive とも WAV bytes を対象にする。
- `metadata_sha256` は archive の publish 済み metadata JSON bytes を対象にし、metadata JSON 自体には hash fields を入れない。

この検証は source text や Markdown string の検査ではなく、service call、file bytes、metadata content、returned `AudioClipRef` を通じて行う。

## 7. 現在の代表検証

確認済みコマンド:

```bash
colcon build --packages-select fa_interfaces fa_audio_window fa_audio_mcp fluent_audio_system
colcon test --packages-select fa_interfaces fa_audio_window fa_audio_mcp --event-handlers console_direct+
```

結果:

- build passed。4 packages finished。
- `fa_interfaces`: 4 passed。
- `fa_audio_window`: 3 test binaries passed。
- `test_export_service_contract`: 11 tests。filesystem store success / conflict / fail-closed を含む。
- `fa_audio_mcp`: 80 passed。

補助確認:

- host `git diff --check`: success。
- host `py_compile`: success。
- host `test_responses.py`: 6 passed。
- Docker/VLAbor ROS2 `test_combined_owner_mcp_launch_smoke.py`: 2 passed in 11.57s。

この検証は FluentAudio child repo 内の Docker/VLAbor ROS2 build、package tests、MCP adapter formatter、combined launch smoke の検証である。親 repo の Agent Runtime tool orchestration、親 VLAbor profile integration、World Station artifact 生成、実 R2/S3 client upload、実 device / model provisioning の検証ではない。
