# fa_voice_command_router アルゴリズム詳細説明書

## 処理手順

```text
command message
  ↓
parse command
  ↓
validate mode / action
  ↓
update router state
  ↓
publish state
  ↓
optional: call TTS service / publish output stop
```

## command handling

command は router の状態遷移 request として扱う。音声 waveform や transcript の解釈はこの node の外で行う。
受理する command は `start`、`stop`、`mode <name>`、`status` のみである。`on` / `off` / `enable` / `disable` / `set_mode` / `mode:<name>` のような互換 alias は持たない。

代表的な状態:

- `standby`
- `command`
- `dictation`
- `mute`

## startup validation

node は起動時に runtime default を使わず、launch / system config から渡された parameter を型付きで読む。必須 parameter の欠落、型違い、空文字、空または重複した `allowed_modes` は fail closed とする。

runtime parameter update は `active`、`mode`、`allowed_modes` だけを受理する。topic / service 名の変更は publisher / client の再生成を伴うため、この node の runtime update では扱わない。

## frame / buffer 前提

音声 frame は扱わない。

## モデル入力形式

モデル入力はない。

## known limitations

- command schema が `std_msgs/msg/String` の場合、構造化 command validation が弱い。将来、専用 message へ移す場合は呼び出し元も同時に更新する。
- router は安全停止 policy の最終判断を持たない。安全 policy は `fa_safety_policy` など app layer の別 node で扱う。
