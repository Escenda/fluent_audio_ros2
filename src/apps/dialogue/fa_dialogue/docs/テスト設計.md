# fa_dialogue テスト設計

## Unit Test

- `FA-DIALOGUE-STATE-001`: valid wake で deterministic session id と `active=true` context を生成する。
- `FA-DIALOGUE-STATE-002`: active session 中の再 wake で新しい `user_turn_id` を生成し、前 turn を置き換える。
- `FA-DIALOGUE-STATE-003`: active turn に一致する `TurnEnd.is_end=true` で `active=false` context を生成する。
- `FA-DIALOGUE-STATE-004`: active turn に一致する ASR FINAL / TIMEOUT / ERROR で `active=false` context を生成する。
- `FA-DIALOGUE-STATE-005`: turn 未開始、session mismatch、turn mismatch は無視する。
- `FA-DIALOGUE-STATE-006`: stale wake と未許可 zero stamp wake を拒否する。
- `FA-DIALOGUE-CONFIG-001`: default config が最小 turn context contract と明示 QoS を持つ。

## Integration Test

- `test_fa_dialogue_ros_graph_publishes_active_then_inactive_turn_context`: real `fa_dialogue_node` を `ros2 run` で起動し、`WakeWordResult` publish 後に `TurnContext(active=true)` を観測する。その後、`AsrResult.STATUS_FINAL` と `TurnEnd.is_end=true` の両経路で `TurnContext(active=false)` を観測する。
- `test_fa_dialogue_launch_uses_config_file_and_publishes_turn_context`: `ros2 launch fa_dialogue fa_dialogue.launch.py config_file:=<tmp_yaml>` で起動し、launch 経由で渡した `config_file` の topic 設定に従って `WakeWordResult` publish 後の `TurnContext(active=true)`、`AsrResult.STATUS_FINAL` 後の `TurnContext(active=false)` を観測する。

検証結果:

- Docker/ROS smoke: `PYTEST_DISABLE_PLUGIN_AUTOLOAD=1 python3 -m pytest src/apps/dialogue/fa_dialogue/test/integration/test_fa_dialogue_ros_graph_contract.py -q` -> `3 passed in 1.65s`
- local non-ROS package tests: `PYTHONPATH=src/apps/dialogue/fa_dialogue pytest -q src/apps/dialogue/fa_dialogue/test` -> `13 passed, 1 skipped`

この integration smoke が証明する範囲は、`fa_dialogue` の turn-context pub/sub 契約のうち、WakeWord -> active `TurnContext`、ASR terminal / TurnEnd terminal -> inactive `TurnContext`、および `ros2 launch` で渡した `config_file` が `fa_dialogue_node` の topic 設定へ反映されることまでである。reasoning、TTS、safety policy、external dialogue backend、robot command proposal、full SO101 model provisioning、親 VLAbor profile integration は検証していない。

## Launch Test

`fa_dialogue.launch.py` の launch smoke は、`config_file:=<tmp_yaml>` を `ros2 launch` に渡し、その設定で指定した topic を使って active / inactive `TurnContext` を観測する。現時点で launch 経路として確認済みなのは config file routing と ASR FINAL 経路の turn-context publish である。QoS policy の個別条件、topic remap の網羅、reasoning、TTS、safety policy、external dialogue backend、robot command proposal、full SO101 model provisioning、親 VLAbor profile integration はこの smoke の検証範囲外である。
