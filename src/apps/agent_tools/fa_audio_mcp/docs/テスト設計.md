# fa_audio_mcp テスト設計

## 方針

テストは tool adapter の振る舞いを証明します。source string や Markdown 文言の存在確認は、この package の正しさの証明として扱いません。

現行の package-local tests は、pure unit test、fake service owner を使う ROS adapter smoke、実 owner node を起動する runtime smoke、installed executable を streamable-http MCP transport 経由で呼ぶ transport smoke に分かれます。ROS adapter smoke は package 内で fake service owner を立て、`FastMCP` tool から `RosAudioTimelineClient`、`rclpy` client、生成済み `fa_interfaces` service 型までを実際に通します。

実 owner node smoke は `ros2 run` で `fa_asr_node` と `fa_audio_window_node` を起動します。ただし、これは `fluent_audio_system` の combined profile launch smoke ではありません。installed transport smoke は別 test として、`ros2 run fa_audio_mcp fa_audio_mcp_server` で起動した server に streamable-http MCP client で接続します。

## Test IDs

- `FA-MCP-SPEC-001`: numeric `<start_unix_ns>..<end_unix_ns>` range を受け付ける。
- `FA-MCP-SPEC-002`: 解決後の `start_unix_ns >= end_unix_ns` を拒否する。
- `FA-MCP-SPEC-003`: export / archive / transcribe は別 scope mapping を使う。
- `FA-MCP-SPEC-004`: unsupported scope、未設定 scope、解決不能な default scope を fail closed にする。
- `FA-MCP-SPEC-005`: archive request は trim 後に空の `reason` を拒否する。
- `FA-MCP-SPEC-006`: export / archive request は format 未指定時に `pcm_s16le` / `wav` / `audio/wav` を明示 request 値にする。
- `FA-MCP-SPEC-007`: `success=false` response を tool error に変換する。
- `FA-MCP-SPEC-008`: 成功 response は service response の resolved range と tool input の requested range を両方保持する。
- `FA-MCP-SPEC-009`: transport、port、timeout、service name、scope mapping の環境変数を検証する。
- `FA-MCP-SPEC-010`: service unavailable / timeout を tool error にする。
- `FA-MCP-SPEC-011`: omitted / null / blank `audio_scope` は tool ごとの explicit default scope key でのみ解決する。
- `FA-MCP-SPEC-012`: `now[-duration]..now[-duration]` range を ROS node clock で numeric range に解決してから service request に渡す。
- `FA-MCP-SPEC-013`: `export_audio_window` は export service / export scope mapping を使い、archive reason や related artifact id を持たない。
- `FA-MCP-SPEC-014`: `FastMCP` tool 呼び出しは、実 `rclpy` client と生成済み service 型を通して ROS service request / response に到達する。
- `FA-MCP-SPEC-015`: 同じ ROS client node の service call は直列化し、複数 executor から同時に同一 node を spin しない。
- `FA-MCP-SPEC-016`: 実 `FastMCP` + `rclpy` 経路でも、service `success=false` は正常 result ではなく MCP tool error になる。
- `FA-MCP-SPEC-017`: in-process `FastMCP` tool が実 `fa_asr_node` / `fa_audio_window_node` に接続し、deterministic ASR / archive frame から transcript、model ref、window ref、WAV clip、metadata、unsupported scope error を返す。
- `FA-MCP-SPEC-018`: installed `fa_audio_mcp_server` executable を streamable-http MCP transport で起動し、numeric time range で実 owner node に対する `transcribe_audio` / `archive_audio_window` tool call と unsupported scope の MCP error result を確認する。
- `FA-MCP-SPEC-019`: installed `fa_audio_mcp_server` executable の streamable-http MCP transport で、`transcribe_audio` / `archive_audio_window` / `export_audio_window` の ROS node clock による `now-10s..now` 解決を確認する。
- `FA-MCP-SPEC-020`: `FLUENT_AUDIO_TIME_MARKERS` 由来の marker endpoint range を MCP adapter 境界で numeric range に解決し、未設定 / 不正 marker は service を呼ばず fail closed にする。

## ROS Adapter Smoke

ROS adapter smoke では次を確認します。

- `FastMCP.call_tool()` から `export_audio_window` / `archive_audio_window` / `transcribe_audio` tool を実行する。
- tool 入力が request builder と scope resolver を通り、生成済み `fa_interfaces` service request に入る。
- `RosAudioTimelineClient` が tool 用 client node の ROS context で service call を spin する。
- fake service owner が受け取った `time_range_spec`、`audio_scope`、`reason`、`related_artifact_ids` が期待値と一致する。
- service response の `AudioClipRef`、`AudioWindowRef`、`AudioModelRef`、`ResolvedTimeRange`、`TranscriptSegment` が tool result JSON に変換される。
- service `success=false` の error code / message が MCP tool error として呼び出し側へ残る。

この smoke は実 owner node のアルゴリズムを証明しません。adapter が ROS graph 上の service 契約を正しく呼び出すことだけを証明します。

## Real Owner Node Runtime Smoke

`test/test_real_owner_graph_smoke.py::test_mcp_tools_call_real_asr_and_audio_window_owner_nodes` では次を確認します。

- `ros2 run fa_asr fa_asr_node` と `ros2 run fa_audio_window fa_audio_window_node` で実 owner node を起動する。
- deterministic `FLOAT32LE` ASR frame と `PCM16LE` archive frame を publish する。
- in-process `FastMCP` tool を real `RosAudioTimelineClient` 経由で呼び、`transcribe_audio` / `archive_audio_window` service に到達する。
- transcript、model ref、audio window ref、WAV clip body、archive metadata が tool result と保存物に反映される。
- owner node が返す unsupported scope error が MCP tool error として呼び出し側へ残る。

この smoke は Docker 内で `1 passed` として確認済みです。ただし、profile-level combined launch と installed `fa_audio_mcp_server` executable / transport は別 smoke の対象です。親 repo Agent Runtime、実 R2/S3 client upload / World Station evidence linkage は未検証です。

## Installed Streamable-HTTP Transport Smoke

`test/test_streamable_http_transport_smoke.py::test_installed_server_streamable_http_transport_calls_real_owner_nodes` では次を確認します。

- `ros2 run fa_asr fa_asr_node` と `ros2 run fa_audio_window fa_audio_window_node` で実 owner node を起動する。
- deterministic `FLOAT32LE` ASR frame と `PCM16LE` archive frame を publish する。
- installed server を `ros2 run fa_audio_mcp fa_audio_mcp_server` で起動する。
- MCP streamable-http transport で `/mcp` に接続する。
- numeric time range で `list_tools`、`transcribe_audio`、`archive_audio_window` を呼び、transport 越しに実 owner node の service behavior へ到達することを確認する。
- unsupported transcribe scope が正常 result ではなく MCP error result になることを確認する。

`test/test_streamable_http_transport_smoke.py::test_installed_server_streamable_http_resolves_now_relative_transcribe_audio` では次を確認します。

- installed server を `ros2 run fa_audio_mcp fa_audio_mcp_server` で起動する。
- `transcribe_audio`、`archive_audio_window`、`export_audio_window` で、`use_sim_time` と `/clock` による ROS node clock の simulated now `30s` から、`time_range="now-10s..now"` を `20s..30s` の numeric range に解決する。
- 実 owner node は解決済み numeric range を受け取り、relative-time support は `fa_asr` / `fa_audio_window` ではなく MCP adapter 境界に閉じることを確認する。

この file-level smoke が証明するのは、installed executable、streamable-http MCP transport、numeric tool-call path、ROS node clock による relative-time 解決、tool adapter、ROS service client、実 `fa_asr_node` / `fa_audio_window_node` owner service の接続が成立することです。親 repo の Agent Runtime / MCP client 統合、自然言語 resolver、turn marker / artifact resolver、実 R2/S3 client upload、World Station evidence linkage、SO101 profile merge / device provisioning は検証しません。

今回の marker resolver 更新後の Docker/VLAbor ROS2 package test では、`fa_audio_mcp` は 101 passed として確認済みです。marker resolver の package-local tests は `FLUENT_AUDIO_TIME_MARKERS` config validation、marker endpoint / offset 解決、未設定 / 不正 marker の fail-closed、request builder 境界での numeric `time_range_spec` 化、MCP tool 境界で service 呼び出し前に marker spec を解決する経路を対象にします。`AudioClipRef.metadata_uri` / `content_sha256` / `metadata_sha256` formatter も package-local response tests の対象です。installed streamable-http relative-time smoke は `PYTEST_DISABLE_PLUGIN_AUTOLOAD=1 python3 -m pytest src/apps/agent_tools/fa_audio_mcp/test/test_streamable_http_transport_smoke.py -q` で `2 passed in 9.35s` として確認済みです。
