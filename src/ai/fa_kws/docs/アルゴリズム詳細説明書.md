# fa_kws アルゴリズム詳細説明書

1. node 起動時に `audio_topic` / `vad_topic` / `output_topic` / `expected_source_id` / `expected_stream_id` が非空であることを検証する。
2. `expected_stream_id` が ROS topic 名と衝突しないことを検証する。
3. `AudioFrame.source_id == expected_source_id` かつ `AudioFrame.stream_id == expected_stream_id` であることを検証する。
4. `AudioFrame` が mono / `FLOAT32LE` / bit depth 32 / normalized [-1.0, 1.0] / `target_sample_rate` と一致することを検証する。
5. 不一致の frame は backend に渡さず reject する。
6. `VadState.source_id == expected_source_id` かつ `VadState.stream_id == expected_stream_id` であることを検証する。
7. 最新の `VadState.probability` を読む。
8. `VadState` が未受信、identity 不一致、`vad.max_age_ms` より古い、または probability が `[0.0, 1.0]` の有限値でない場合は frame を処理せず skip する。
9. probability が `vad.probability_gate` 未満の場合は backend を呼ばず skip する。
10. backend `process()` に samples, sample rate, VAD probability, timestamp を渡す。
11. backend は防御的に VAD gate を再評価し、gate 未満なら sherpa-onnx stream に波形を渡す前に stream を reset して `nullopt` を返す。
12. detection があれば `WakeWordResult` を publish する。

VAD probability は gating signal です。`vad.probability_gate` は `(0.0, 1.0]` の有限値だけを受け付け、`0.0` や負値を「gating disabled」として扱いません。VAD 推定は `fa_vad` に分離します。
Resampling / downmix / bit-depth conversion は `fa_kws` では行いません。必要な変換は upstream の format/processing node で明示します。
`source_id` / `stream_id` は音声 identity です。`expected_source_id` と `expected_stream_id` は system config で明示し、`audio_topic` は ROS transport の接続点として扱います。これにより、同じ topic に誤って流れ込んだ別 source や別 stream の音声を KWS backend へ混入させません。
`VadState` も同じ identity を持つため、KWS は別 stream の VAD probability を現在 stream の gate として流用しません。不一致 state を受けた場合は最新 VAD state を clear し、次の valid state まで audio frame を処理しません。
音声記録、debug dump、WAV encode、PCM16 conversion は `fa_kws` の責務ではありません。後段確認用に音声を保存する場合は `fa_record` を pipeline に追加します。
`kws_wav_tool` も同じ責務境界に従います。tool 入力 WAV は `--sample_rate` と一致する mono / IEEE float32 / normalized [-1.0, 1.0] でなければならず、tool 内で resample、downmix、bit-depth conversion は行いません。
`kws_wav_tool` は VAD topic を購読しない offline 評価 tool なので、起動時に `--assume-vad-speech` を必須にし、backend には `vad_prob=1.0` と `vad_threshold=1.0` を明示して渡します。backend 側の VAD gate を `0.0` で無効化する経路は持ちません。
また、tool 側でも backend numeric config は CLI から明示的に受け取り、`SherpaOnnxKwsBackendConfig` の全 field を初期化してから backend を生成します。未初期化値や hidden default による engine 起動は fail-closed 境界を壊すため禁止です。
backend name validation と concrete backend construction は `backends/factory` で行います。node は `KwsBackendSettings` を組み立て、`KwsBackend` interface だけを保持します。
sherpa-onnx provider は `backend.execution_provider` で明示し、空値や未対応 provider は backend constructor 内の config validation で reject します。
sherpa-onnx reference worker の検出結果は confidence score を返さないため、backend は binary detection score `1.0` を明示的に返します。
