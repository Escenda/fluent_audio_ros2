# fa_asr アルゴリズム詳細説明書

1. node 起動時に `expected_stream_id` が ROS topic 名と衝突しないことを検証する。
2. `AudioFrame` が data を持ち、configured source / stream identity、mono / `FLOAT32LE` / 32-bit / normalized [-1.0, 1.0] / `target_sample_rate` と一致することを検証する。
3. 不一致の frame は backend に渡さず reject する。
4. `VadState.source_id == expected_source_id` かつ `VadState.stream_id == expected_stream_id` であることを検証する。
5. VAD active 中の samples を buffer に追加する。
6. VAD end または timeout で buffer を finalize する。
7. backend に samples と sample rate を渡す。non-streaming command backend では `backend.result_format` に従う output を transcript segments へ変換し、streaming backend では backend 固有 protocol の stream result を扱う。
8. transcript を `AsrResult` として publish する。
9. `TranscribeAudio` request を受けた場合は、numeric `time_range_spec` を parse し、ASR-ready rolling timeline 上で requested `[start,end)` を完全に覆う最小の sample-boundary `[actual_start,actual_end)` へ量子化して slice する。
10. ASR-ready rolling timeline が空なら `window_not_found`、requested range または quantized selected range が retention window 外なら `range_outside_window`、保持済み frame はあるが gap により selected range を連続 coverage できなければ `range_not_continuous` を返す。
11. timeline failure、unsupported `audio_scope` では backend を呼ばず、空 clip、latest frame、`fa_audio_window` の archive/export window へ fallback しない。
12. backend timeout/error/invalid output/empty transcript の場合は `success=false` と明示 error code を返す。
13. 成功時は transcript segment、explicit `timeline.*` parameter 由来の `AudioWindowRef`、backend model ref、resolved time range を返す。

Python version / venv が異なる ASR engine は node 内 import ではなく external worker / process / container backend として扱います。
Resampling / downmix / bit-depth conversion は `fa_asr` では行いません。必要な変換は upstream の format/processing node で明示します。
`source_id` は `expected_source_id` と一致し、`stream_id` は `expected_stream_id` と一致する必要があります。別 source / stream の frame を同じ ASR buffer に混在させません。
VAD state も同じ identity と照合し、別 stream の VAD end で現在の ASR buffer を finalize しません。

## 根拠分類

この説明書では、公式一次情報、local 実測、FluentAudio の設計判断を分けて扱います。
NIM / Riva / NGC / NeMo docs の記述は model family や serving stack の根拠であり、local `.nemo` worker の実行成功を代替しません。
逆に、local `.nemo` の offline transcription 成功は、その model file と local NeMo process の full-context API が使える証跡であり、NIM / Riva serving や NeMo cache-aware streaming の成功証明ではありません。

## 2026-05-22 調査で固定する backend 境界

この節は、NGC / Riva / NIM / gRPC / local NeMo / Parakeet streaming を、backend algorithm 上の証拠段階として 1 個ずつ切り分けるためのものです。
同じ Parakeet 1.1B RNNT Multilingual という名前が出ていても、どの境界で何が成立したのかを混ぜません。

### 1. NGC `download-version` は artifact acquisition だけである

`ngc registry model download-version <target>` は、NGC registry にある指定 model version を local disk へ取得する command です。
official CLI docs 上の `<target>` は `org/[team/]model_name[:version]` 形式で、version を省略すると latest が対象になります。
FluentAudio では再現性のため version を省略しません。

`--dest`、`--file`、`--exclude`、`--format_type` は取得先、取得対象 file の絞り込み、除外 wildcard、CLI 出力形式を制御する option です。
これらは artifact acquisition の option であり、次を行いません。

- NIM container を起動する。
- Riva server / Triton / RMIR を起動する。
- `.riva` / Riva artifact を local NeMo streaming 用に変換する。
- `.nemo` の config を書き換える。
- finite attention context を追加する。
- `ASRModel.restore_from(...)` を実行する。
- `nemo_rnnt_streaming_worker health` を実行する。
- `nemo_offline_transcribe_worker health` を実行する。
- ASR transcript が non-empty になることを証明する。

したがって、NGC download 成功は「pinned artifact contents が local path に存在する」ことだけを意味します。
backend readiness は、その後の local file integrity、restore、capability validation、worker health、speech fixture、full ROS graph validation で別々に判定します。

`prepare_nemo_offline_transcribe_asr` は、この artifact acquisition と env 出力を `nemo_offline_transcribe` backend 向けに固定する local preparation script です。
default model id は `parakeet-1.1b-multilingual-offline`、artifact は `nvidia/riva/parakeet-rnnt-riva-1-1b-unified-ml-cs-universal:trainable_v1.0`、expected file は `Parakeet-RNNT-XXL-1.1b_merged_universal_spe8.5k_1.0.nemo` です。
script は NIM / Riva server / gRPC / Docker / model server を起動せず、runtime ASR も行いません。
supported id / artifact、expected file、worker executable を検証し、host と `/ros2_ws/src/fluent_audio_ros2` container target 用の sourceable env と trace を出すだけです。

### 2. Riva/NGC artifact を local `.nemo` input に使うことは Riva/NIM serving と同義ではない

現在の実験で参照している artifact は次です。

```text
nvidia/riva/parakeet-rnnt-riva-1-1b-unified-ml-cs-universal:trainable_v1.0
```

この artifact 由来の `.nemo` を local disk に置き、FluentAudio の local worker が `ASRModel.restore_from(...)` で読むことはできます。
しかし、これは Riva server や NIM container を使っているという意味ではありません。

Riva/NIM serving では、server/container 側が model loading、optimization、Triton/TensorRT execution、serving profile、request/response schema、streaming state を所有します。
local NeMo worker では、worker process が `.nemo` restore、Torch/CUDA runtime、model capability validation、JSONL / command protocol、stderr/stdout separation、temporary WAV bridge、cache state を所有します。
同じ artifact 名でも、実行境界が違うため、成功条件も違います。

### 3. NIM/Riva support matrix は serving-stack capability の根拠であり local worker success ではない

NVIDIA ASR NIM docs は、ASR NIM を pre-trained NeMo model と TensorRT / Triton inference stack を container に package した service として説明しています。
NIM の streaming mode は、audio arrival に応じて partial transcripts を返す serving mode です。

NIM support matrix では、Parakeet 1.1B RNNT Multilingual は `CONTAINER_ID=parakeet-1-1b-rnnt-multilingual` で選択され、`NIM_TAGS_SELECTOR` によって `mode=ofl`、`mode=str`、`mode=str-thr`、`mode=all` などの profile を選びます。
同 matrix では `ja-JP` は default / prompt type の supported language として扱われます。

この情報が証明するのは、NIM/Riva serving stack と指定 profile 上の capability です。
証明しないものは次です。

- local `.nemo` が `ASRModel.restore_from(...)` で読めること。
- local NeMo worker が RNNT / Transducer capability を満たすこと。
- local worker が finite attention context を持つこと。
- local worker が cache-aware streaming step を進められること。
- FluentAudio JSONL protocol が `health_ok` / `start` / `audio` / `finish` に成功すること。
- ROS graph 上で `AsrResult` の non-empty final が publish されること。

### 4. local `nemo_rnnt_streaming` は health-first の fail-closed backend である

local `nemo_rnnt_streaming` backend は、Riva/NIM の代替名ではありません。
この backend は、local `.nemo` を NeMo runtime で restore し、cache-aware streaming を worker process 内で進める backend です。
完了扱いするには、最低限次を順番に満たす必要があります。

1. local `.nemo` path が readable で、version / size / checksum が期待 artifact と整合する。
2. `ASRModel.restore_from(...)` が成功し、model が eval mode へ入る。
3. model が RNNT / Transducer 系である。
4. model sample rate、channel contract、language metadata が backend config と矛盾しない。
5. encoder が cache-aware streaming に必要な API を持つ。
6. requested finite attention context が supported contexts に含まれる。
7. streaming params を確立できる。
8. `health_ok` response が backend config と一致する。
9. `start` が stream-local cache state を初期化できる。
10. `audio` が base64 float32le chunk を受理し、cache-aware step を進められる。
11. partial / final transcript が protocol 上 valid で、speech fixture では non-empty text が返る。
12. full ROS graph で ASR-ready stream、control window、backend invocation、result publish が成立する。

この chain のどこかが欠ける場合は fail closed です。
download 成功、restore 成功、support matrix、`audio_accepted`、empty final は、この chain の後段を代替しません。

### 5. offline/full-context と true cache-aware streaming は別 backend である

`nemo_offline_transcribe` は complete audio を NeMo offline / full-context `transcribe(...)` API に渡す backend です。
これは `start` / `audio` / `drain` / `finish`、encoder cache、previous hypothesis、partial result、low-latency commit を持ちません。

offline success は重要です。
local `.nemo` が NIM/Riva/gRPC なしに restore でき、NeMo full-context API が non-empty text を返せることを示します。
しかし、それは low-latency streaming の証明ではありません。
NeMo docs には full-context model を simulated streaming 的に扱う方法がありますが、accuracy / latency tradeoff と training/inference mismatch を持つ別 policy です。
FluentAudio では、これを `nemo_rnnt_streaming` の成功条件を弱める形では使いません。
必要なら別 backend / 別 policy として設計します。

### 7. backend docs の書き方

backend algorithm details では、次を同じ表に潰しません。

- artifact acquisition
- local file integrity
- model restore
- health readiness
- speech transcript smoke
- cache-aware streaming state machine
- ROS graph validation
- accuracy evaluation

どれか 1 つが通っても、後段が通ったとは書きません。
逆に、ある段階が fail closed した場合は、そこで止めたことを正しい状態として記録します。
FluentAudio では、成功していないものを成功に見せるより、まだ通っていない境界を正確に残すことを優先します。

## NVIDIA 系 runtime 境界の分類

NIM、Riva、gRPC、NGC artifact、local `.nemo` + NeMo は同じものではありません。
ここを混ぜると、公式 docs の「その serving stack では対応している」という情報を、FluentAudio の local backend が実行済みであるかのように誤読します。

### NIM

NIM は GPU accelerated Docker container として配布される service 境界です。
application は model file を直接 restore せず、gRPC / HTTP API で NIM container に request を送ります。
container 側が model loading、GPU execution、batching、streaming state を内部で持ちます。

FluentAudio が NIM を使うなら、backend は「NIM API client backend」です。
その場合の検証対象は container readiness、API schema、gRPC / HTTP timeout、server-side model selection、network failure、streaming response handling です。
local `.nemo` を Python process で `ASRModel.restore_from(...)` する backend と同じ完了条件にはしません。

### Riva

Riva は server / model repository / RMIR / Triton / gRPC client を含む speech serving stack です。
Riva ASR は offline と streaming の両方を扱い、streaming recognition では server が intermediate transcript を返せます。
Riva server は request の `RecognitionConfig` 等に基づき、deployed model を選択できます。
Riva の multilingual model table に `ja-JP` が含まれていても、それは Riva serving stack 上の capability であり、FluentAudio local NeMo worker の実行成功を代替しません。

FluentAudio が Riva を使うなら、backend は「Riva gRPC backend」です。
その場合、model repository / RMIR / server config / client request / server response / gRPC failure が検証対象になります。
local `.nemo` worker や command process backend に Riva の責務を混ぜません。

### gRPC

gRPC は通信方式であり、NIM や Riva そのものではありません。
NIM / Riva は gRPC を API protocol として使えますが、FluentAudio で gRPC を使う場合は backend protocol の一種として定義します。
gRPC backend では、channel readiness、deadline、request/response schema、stream cancellation、server error mapping が contract になります。
local command backend の stdin/stdout、JSONL、temporary file contract とは別です。

### NGC artifact

NGC artifact は model artifact の取得元です。
`download-version` は指定 version の contents を local disk に置く操作であり、model restore、format conversion、NIM/Riva server 起動、worker health、ASR transcript 成功を行いません。
artifact metadata、size、SHA256 は selection / integrity の根拠です。
runtime 成功は、実際の backend process がその artifact を読み、必要な API と入力契約を満たしたときだけ確認できます。

### local `.nemo` + NeMo

local `.nemo` + NeMo は Python process 内で `ASRModel.restore_from(...)` を実行し、restore 済み model object に `model.transcribe(...)` または cache-aware streaming step を呼ぶ方式です。
この方式では、NIM/Riva server は存在せず、model loading、Torch/CUDA runtime、sample rate、language metadata、worker protocol、temporary file bridge、stderr/stdout separation を worker が直接所有します。

local `.nemo` でも offline と streaming は別です。
offline/full-context は complete audio を渡して transcript を得る処理です。
cache-aware streaming は limited right context、stream-local cache state、chunk/shift size、previous hypothesis、partial/final semantics を持つ別の処理です。
offline/full-context model を chunking して疑似 streaming にできる場合があっても、compute duplication、training/inference mismatch、latency/accuracy tradeoff があるため、FluentAudio では低遅延 streaming 成功とは扱いません。

## streaming backend の共通アルゴリズム

streaming ASR backend は、non-streaming command backend とは別の状態機械として扱います。
`fa_asr` は VAD / TurnContext / trigger contract によって ASR window を開閉し、backend はその window に対応する stream session を持ちます。

処理順序は次です。

1. node 起動時、backend の `health` を実行して model / worker capability を検証する。
2. `TurnContext.active=true` かつ非空 `session_id` / `user_turn_id` を受け取るまで、ASR result を commit しない。
3. control input が `open_on` 条件を満たしたとき、ASR control window を開く。
4. control window open と同時に backend stream を `start` する。
5. window 中に受け取った ASR-ready `AudioFrame` を sample boundary のまま backend stream へ `audio` として push する。
6. backend から partial が返れば `AsrResult.STATUS_PARTIAL` と `AsrEvent.EVENT_STREAM_PARTIAL_RESULT` を publish する。
7. control input が `close_on` 条件を満たしたとき、window を閉じる。
8. `submit_on_close=true` の場合は backend stream を `finish` し、final transcript を `AsrResult.STATUS_FINAL` として publish する。
9. backend が timeout/error を返した場合は `AsrResult.STATUS_ERROR` と `AsrEvent.EVENT_FAIL_CLOSED` を publish し、別 backend へ切り替えない。

streaming backend の `audio_accepted` は transcript 成功ではありません。
これは「worker が audio chunk を stream state に受理した」ことだけを意味します。
partial / final transcript は別 event として扱います。
`finish` が返っても、final text が空で、prior non-empty partial も無い場合は ASR 成功ではありません。
transport state machine が閉じたことと、speech content を認識できたことを分けます。

Riva / NIM の streaming は server 側が stream state を所有します。
local NeMo cache-aware streaming は worker process が cache state を所有します。
どちらも `fa_asr` node 本体が audio を勝手に変換したり、未対応 backend へ fallback したりする理由にはなりません。

## ASR backend 別アルゴリズム境界

`fa_asr` の backend は、同じ `AsrBackend` / `StreamingAsrBackend` contract の下にありますが、実行形態は同じではありません。
ここを混ぜると、OpenAI API、Whisper wrapper、local command、NeMo streaming worker がすべて同じものに見えてしまい、失敗時にどこを疑うべきか分からなくなります。

共通原則は次です。

1. `fa_asr` は ASR-ready audio だけを backend に渡す。
2. backend は自分が受け取れる形式を明示する。
3. 未対応形式を受け取ったら変換せず reject する。
4. backend output は明示された result format / JSONL protocol に従って検証する。
5. 失敗時に別 backend へ fallback しない。

### command process 共通層

`local_command`、`whisper.cpp`、`parakeet_worker`、`nemo_offline_transcribe`、`openai_realtime`、`openai_transcriptions` は、外部 command process を呼び出す系統の backend です。
これらは model engine、SDK、network API、NeMo/Torch runtime を `fa_asr` node 内へ import しません。
`fa_asr` は ASR-ready audio と backend config を検証し、worker process の入出力 contract を守ります。

処理 phase は次です。

1. config validation
   - `backend.command` が executable として解決できることを確認する。
   - `backend.result_format` が `plain_text` または `segments_json_v1` として明示されていることを確認する。
   - `backend.args` に `{audio}`、`{model}`、`{sample_rate}` が含まれることを確認する。
   - model path / model id / API key env など、backend 固有の必須値を確認する。
2. optional / required health check
   - `health_args` が指定されている backend は、起動時に command を実行して model / credential / executable contract を確認する。
   - health check が timeout / non-zero exit の場合、backend 起動完了扱いにしない。
   - health check は file existence だけではなく、backend ごとの runtime dependency や credential / model load readiness を確認する gate として扱う。
3. request validation
   - `AsrRequest.payload` が mono `FLOAT32LE`、正の sample rate、非空、finite、normalized `[-1.0, 1.0]` であることを確認する。
   - PCM16 / WAV / stereo / 別 sample rate を受け取っても、ここで変換しない。
4. audio serialization
   - validated samples を `float32le_raw` の一時 `.f32` file に書く。
   - file は backend command の入力 path であり、音声形式変換の中間生成物ではない。
   - backend worker が model API bridge として WAV / tensor / provider-specific payload に変換する場合、その変換は backend 境界内で明示し、`fa_asr` node 本体の hidden DSP と混同しない。
5. command execution
   - `{audio}`、`{model}`、`{sample_rate}`、必要に応じて `{language}` / `{output}` を argv に展開する。
   - timeout、non-zero exit、stderr reason を明示的に扱う。
6. output parsing
   - `plain_text` は非空 text を selected request samples 全体の 1 segment として扱う。
   - `segments_json_v1` は top-level が `result_format` と `segments` だけの strict JSON でなければならない。
   - segment は integer `start_sample` / `end_sample`、非空 `text`、任意の非空 `speaker_label` だけを許可する。
   - invalid JSON、schema 不一致、range 不正、overlap、unsorted order、空 text は fail closed する。

この共通層は、backend engine の正しさを証明しません。
証明するのは「`fa_asr` が渡す audio と worker output contract が壊れていないこと」です。
実 model / external API / real worker smoke / full ROS graph validation は backend ごとに別途必要です。

### local_command

`local_command` は、任意の local executable を ASR backend として接続するための最小 command backend です。
状態は実装済みです。ただし、この backend は engine generic なので、個別 ASR engine の精度や model capability を証明しません。

必須の境界は次です。

- model は `backend.model_path` として local readable file を要求する。
- engine 固有の import、Python version、venv、GPU runtime は command の内部責務である。
- `fa_asr` は raw `.f32` path と sample rate を渡すだけで、command 内部の model loading や decoder algorithm を知らない。
- `health_args` が指定された場合だけ startup health check を行う。profile で本番利用するなら health check を明示する。

`local_command` は汎用 backend であり、Whisper や Parakeet の成功条件を暗黙に知りません。
engine 固有の制約を持つ場合は、専用 backend docs に記載します。
unsupported input、empty transcript、invalid `segments_json_v1`、timeout、non-zero exit は fail closed です。

### whisper.cpp

`whisper.cpp` backend は、`whisper.cpp` wrapper / worker を subprocess として呼び出します。
状態は実装済みです。

重要な点は、`fa_asr` が `whisper-cli` の都合で WAV / PCM16 へ変換しないことです。
wrapper は FluentAudio の ASR-ready `float32le_raw` path と sample rate を受け取り、自分の責務として whisper.cpp が扱える形式へ橋渡しします。
この橋渡しは whisper.cpp worker の model API adapter であり、`fa_asr` node 本体の hidden resample / downmix ではありません。

拒否条件は次です。

- model file が無い。
- command が executable ではない。
- `{audio}` / `{model}` / `{sample_rate}` placeholder が欠けている。
- result format が未指定または unsupported。
- worker output が empty transcript / invalid segments である。

Whisper backend が失敗しても、OpenAI、Parakeet、Nemo へ切り替えません。
Whisper model の language support、accuracy、device acceleration は wrapper / model runtime 側の検証対象であり、FluentAudio command contract test だけでは証明しません。

### parakeet_worker

`parakeet_worker` は、Parakeet 系 ASR を外部 worker / process / container として呼び出す command backend です。
これは `nemo_rnnt_streaming` とは別です。
状態は backend slot と command contract として実装済みです。

境界は次です。

- `backend.model` は model id であり、local `.nemo` path ではない。
- `health_args` は必須で、`{model}` を含む startup health check を通す。
- `fa_asr` は Parakeet / NeMo / PyTorch を import しない。
- worker は `float32le_raw` audio と sample rate を受け取り、`backend.result_format` に従って結果を返す。

この backend は command request / response 型です。
streaming session、encoder cache、partial commit、attention context 管理は `nemo_rnnt_streaming` の責務であり、ここへ混ぜません。
NIM/Riva/NGC/local `.nemo` のどれを使うかは worker 側の runtime policy です。
`parakeet_worker` という名前だけで NIM/Riva serving、NeMo restore、streaming capability、Japanese fixture success は証明されません。

### nemo_offline_transcribe

`nemo_offline_transcribe` は、local `.nemo` file を NeMo offline / full-context `transcribe(...)` API で呼ぶ non-streaming command / worker backend です。
working tree 上では backend loader、worker、unit tests、profile work が作成済みです。
PO 検証により real worker health、raw `FLOAT32LE` fixture transcription、opt-in file-source full ROS graph smoke は通過済みです。
ただし、accuracy 評価、`TranscribeAudio` service integration、NIM / Riva / gRPC backend readiness、generic live microphone ASR は未検証であり、backend をそれらまで含む本番相当経路で検証済みとは扱いません。

runtime evidence として、`fluent-audio-runtime` container で Torch `2.11.0+cu130`、CUDA available の状態を確認し、`ASRModel.restore_from(...)` により local `.nemo` を `EncDecRNNTBPEModel` として restore できることを確認しています。
対象 file は `models/nemo_rnnt_streaming/parakeet-rnnt-riva-1-1b-unified-ml-cs-universal_vtrainable_v1.0/Parakeet-RNNT-XXL-1.1b_merged_universal_spe8.5k_1.0.nemo`、size `4011233560`、SHA256 `52332e96ef68ff8cfefd1d8d7b8c5d7b5333faa3cfac87ed4cc7b5ec3d5821c0` です。
PO の直接 NeMo API 証跡として次を使いました。

```text
ASRModel.restore_from(...);
model.transcribe(
    audio=[...],
    batch_size=1,
    return_hypotheses=False,
    num_workers=0,
    verbose=False,
)
```

`/tmp/fluent_audio_asr_fixture/ja-pronunciation-practice5.ogg` に対して、次の non-empty Japanese output が返りました。

```text
天 気 練 習 残 業 安 ん な り 電 波 宣 兵 電 米 宣 本 専 用 本 屋 三 円 単 位
```

この direct API evidence 単体では、offline / full-context local NeMo transcribe の viability だけを示します。
accuracy、`TranscribeAudio` integration 完了、file-source full ROS graph success は証明しません。
また、`nemo_rnnt_streaming` の streaming `health_ok`、`start`、`audio_accepted`、partial / final transcript 成功も証明しません。
この direct API evidence は worker algorithm そのものではありません。
現行 `nemo_offline_transcribe_worker` は runtime signature を調べ、`batch_size`、`return_hypotheses`、`verbose`、および受け付けられる場合の language key だけを `model.transcribe(...)` へ渡します。
worker が `num_workers` を渡すとは記載しません。

algorithm boundary は次です。

1. `fa_asr` は ASR-ready mono `FLOAT32LE` samples と sample rate を backend request として渡す。
2. backend loader は `backend.model_path` が local `.nemo` file であること、`backend.language` が非空であること、`backend.sample_rate_hz` が正の整数であること、`backend.channels == 1` であることを検証する。
3. startup health は worker に `health --model {model} --language {language} --sample-rate <hz> --channels 1` を実行させる。
4. worker health は NeMo ASR module、`ASRModel.restore_from(...)`、model `eval()`、offline `transcribe` method、model sample rate、language metadata を検証する。
5. transcribe request では worker が raw `.f32` file を読み、byte length、empty audio、finite samples、normalized `[-1.0, 1.0]` を再検証する。
6. worker は validated float32 samples を temporary PCM16 mono WAV に書く。
7. この WAV は NeMo offline API が file path input を受けるための model API bridge であり、`fa_asr` node 本体の hidden resample / downmix / sample-format conversion ではない。
8. bridge は sample rate と channel count を変えず、finite / normalized validation 後に value range を PCM16 表現へ写す。音声の意味を改善・補正する DSP、gain、normalization、denoise、VAD ではない。
9. worker は `model.transcribe([wav_path], batch_size=1, return_hypotheses=False, verbose=False, language=<supported if accepted by signature>)` 相当を呼ぶ。worker が追加する language key は `language` / `language_id` / `source_lang` / `source_language` のうち runtime signature が受け付けるものだけです。
10. worker は result が single transcript として取り出せることを検証し、empty text を fail closed にする。
11. output は `plain_text` または strict `segments_json_v1` として emit する。
12. 失敗時に `nemo_rnnt_streaming`、Whisper、OpenAI、NIM/Riva へ fallback しない。

この backend は streaming session、cache state、partial result、`audio_accepted` を持ちません。
full-context model を streaming 成功扱いするための互換 layer ではなく、offline transcription を明示的に扱う別 backend です。
NeMo docs が説明する buffered / chunked streaming は offline model を streaming 的に使う方法ではありますが、compute duplication と training/inference mismatch を持つため、`nemo_offline_transcribe` の成功を low-latency streaming 成功として扱いません。

fail-closed 条件は次です。

- command missing / not executable
- model path missing / unreadable / `.nemo` ではない
- NeMo ASR module または `ASRModel.restore_from(...)` unavailable
- restore failure
- model が `transcribe` を持たない
- model sample rate と configured sample rate の不一致
- model が language metadata を公開し、configured language が含まれない
- `channels != 1`
- raw audio file empty、byte length 不整合、NaN / Inf、normalized range violation
- NeMo `transcribe` exception
- unsupported result shape、複数 transcript、empty transcript
- invalid output schema、timeout、non-zero exit

### openai_realtime / openai_transcriptions

OpenAI 系 backend は、OpenAI API 呼び出しを external worker 側へ閉じ込めます。
`fa_asr` は OpenAI SDK、WebSocket client、network retry、response schema 差分を持ちません。
状態は command backend slot として実装済みです。

共通境界は次です。

- API key は `backend.openai_*.api_key_env` で指定された環境変数から読む。
- worker process には canonical な `OPENAI_API_KEY` として渡す。
- key 値を argv、stdout、stderr、ROS topic、通常 log に出さない。
- API endpoint、request format、rate limit、retry、OpenAI response parsing は worker の責務である。
- `fa_asr` は worker output を `plain_text` または `segments_json_v1` として検証する。

`openai_realtime` という名前であっても、現行の backend 境界は external command process です。
`fa_asr` の streaming session state と同一視しません。
Realtime API を真に streaming session として扱う場合は、`nemo_rnnt_streaming` と同じように `StreamingAsrBackend` として別 contract を定義する必要があります。
API key missing、health failure、worker timeout、network/API error、empty transcript、invalid output schema は fail closed です。
OpenAI API の model capability は worker / provider response に依存するため、FluentAudio の command contract test だけで provider 側成功を証明しません。

## `nemo_rnnt_streaming` backend のアルゴリズム契約

`nemo_rnnt_streaming` は、NeMo RNNT / Transducer model を local `.nemo` file として読み込む streaming 専用 backend です。
`fa_asr` node 本体は NeMo / PyTorch / Parakeet を import せず、model-specific 処理は `nemo_rnnt_streaming_worker` process に閉じ込めます。

`nemo_rnnt_streaming` は JSON Lines protocol で次の command を受けます。

- `health`: `.nemo` restore と capability 検査。
- `start`: stream-local state 初期化。
- `audio`: validated float32 chunk の受理と cache-aware streaming step。
- `drain`: 現時点の partial hypothesis 取得。
- `finish`: stream finalization と final transcript 生成。
- `cancel`: active stream 破棄。

raw worker protocol としては、`start` が prior `health` なしで届いても、worker は restore / capability validation を行ってから stream を開始できます。
`health` 後に `start` が届いた場合、worker は `start` command の config を health 済み config と比較し、不一致なら拒否します。
ただし system 完了条件は health-first です。
`fa_asr` backend としては起動時 `health` で model、sample rate、attention context、streaming params、partial capability を検証し、失敗を runtime audio arrival まで遅延させません。

`audio_accepted` は transcript 成功ではありません。
これは active stream に audio chunk が受理されたことだけを示します。
partial transcript、final transcript、`finished` は別 event です。
`drain` は新しい audio を追加せず、latest non-empty partial がある場合だけ partial を返します。
`finish` は protocol 上 `final` と `finished` を返しますが、final text が空のまま speech window を成功扱いしません。
backend wrapper は final text が空で prior non-empty partial がある場合に latest partial を final commit 候補にできますが、prior partial も無い empty final は transport completion であり、ASR 成功ではありません。
`fa_asr` publish 経路では empty final result 自体が publish され得ます。
代表検証の合格条件は final result の存在だけではなく non-empty text です。

`nemo_rnnt_streaming` は streaming backend です。
現行 `transcribe()` は常に失敗し、`TranscribeAudio` service の non-streaming transcription backend としては使えません。
`TranscribeAudio` service が ASR-ready rolling timeline から sample-boundary slice を作れても、この backend ではその slice を offline/full-context ASR へ渡す contract がありません。
full-context / offline transcribe を行うなら別 backend policy が必要であり、`nemo_rnnt_streaming` の streaming session、cache state、partial/final semantics に混ぜません。

`fa_asr` node が backend へ渡す前に保証する audio contract は次です。

- mono
- `FLOAT32LE`
- normalized `[-1.0, 1.0]`
- finite samples only
- `backend.sample_rate_hz` と一致
- `AudioFrame` timeline 上で連続
- `expected_source_id` / `expected_stream_id` と一致

worker JSONL が直接検証する contract は次です。

- `session_id` が active stream と一致する。
- `encoding == base64_float32le` である。
- `sample_count * channels * 4` と payload byte length が一致する。
- decoded float32 samples が finite である。
- decoded float32 samples が `[-1.0, 1.0]` の範囲内である。

`AudioFrame` timeline、`source_id`、`stream_id` は `fa_asr` node 側の責務です。
worker は ROS message を知らないため、worker 内でこれらを再検証しません。

`nemo_rnnt_streaming` は resample、downmix、PCM16 conversion、WAV decode、value normalization、missing chunk 補完を行いません。
これらが必要な場合は、upstream processing node の責務として明示します。

### backend-adjacent concepts の分離

Parakeet RNNT multilingual を FluentAudio で使う場合、backend 周辺の用語を同一視してはいけません。
NIM、Riva、gRPC、NGC artifact、local `.nemo`、NeMo cache-aware streaming、language profile、worker health、full ROS graph validation はそれぞれ別の証明範囲を持ちます。

#### NIM

NIM は NVIDIA が配布する ASR serving container です。
pre-trained NeMo model と TensorRT / Triton inference stack を self-contained container に package し、model download、optimization、serving profile selection、serving endpoint を container 側で扱います。
ASR NIM の streaming mode は audio arrival に応じて partial transcript を返す serving mode です。

Parakeet RNNT Multilingual の NIM profile は `CONTAINER_ID=parakeet-1-1b-rnnt-multilingual` と `NIM_TAGS_SELECTOR` で選びます。
`mode=ofl`、`mode=str`、`mode=str-thr`、`mode=all`、`diarizer=sortformer`、`vad=silero`、`type=default`、`type=prompt`、`type=indic` は NIM serving profile の選択肢であり、FluentAudio local worker の `backend.*` config ではありません。

FluentAudio の `nemo_rnnt_streaming` backend は NIM container を起動せず、NIM endpoint に接続しません。
NIM の support matrix に streaming support が載っていても、local `.nemo` worker が `health`、`start`、`audio`、non-empty `final` に成功することは証明されません。

#### Riva

Riva は NVIDIA の speech serving / deployment stack です。
Riva server、model repository、Triton、serving pipeline、pre/post-processing、offline / streaming recognition を含む runtime 境界です。

FluentAudio の現行 `nemo_rnnt_streaming` backend は Riva server を起動しません。
Riva model card や Riva artifact metadata は artifact selection の根拠として使えますが、Riva serving 成功や Riva model card の capability は local NeMo worker の実行成功を代替しません。

#### gRPC

gRPC は NIM / Riva service client が使う network protocol 境界です。
Riva Python client は gRPC communication、authentication、audio encoding、request / response serialization を扱います。

現行 `nemo_rnnt_streaming` backend の transport は gRPC ではありません。
`fa_asr` は `nemo_rnnt_streaming_worker` process と stdin / stdout の JSON Lines protocol で通信します。
将来 NIM / Riva を使う場合は、local `.nemo` worker の内部実装として混ぜず、NIM / Riva gRPC backend として別 contract を定義します。

#### NGC artifact / `download-version`

NGC artifact は model file と metadata の配布単位です。
現時点で検証対象にしている artifact は `nvidia/riva/parakeet-rnnt-riva-1-1b-unified-ml-cs-universal:trainable_v1.0` です。
NGC CLI metadata では `trainable_v1.0`、`totalFileCount=1`、`totalSizeInBytes=4011233560`、file path `Parakeet-RNNT-XXL-1.1b_merged_universal_spe8.5k_1.0.nemo`、file SHA256 base64 `UjMulu9o/4z+/R2Ne4xde1Mz+qPPrIftTMe17D1YIcA=`、`NeMo Version=1.23.0`、`Dataset Size=94k hrs`、`Vocabulary Size=8.5k` を確認済みです。

`ngc registry model download-version` は指定 version の contents を local disk に取得する操作です。
`--dest` は取得先 directory を指定するだけであり、artifact を NeMo local streaming 用へ変換せず、`.nemo` config を変更せず、finite attention context を追加せず、worker `health` も実行しません。
version を省略すると NGC 側 latest を対象にするため、FluentAudio の backend 検証では version を pin します。

#### local `.nemo`

local `.nemo` は `nemo_rnnt_streaming_worker` が `ASRModel.restore_from(...)` で直接 restore する checkpoint です。
この file は git 管理せず、host または prepare container で取得し、runtime container には volume として渡します。
runtime node 起動時に NGC download を行う設計にはしません。

local file の存在、file size、SHA256、model card metadata は必要な証跡ですが、ASR 成功の証明ではありません。
local `.nemo` backend の成立には、restore、capability validation、finite attention context validation、streaming params setup、worker protocol、non-empty transcript、full ROS graph validation が別途必要です。

#### cache-aware streaming

cache-aware streaming は、restore した model に raw audio を渡すだけの処理ではありません。
worker は RNNT / Transducer model class、encoder の cache-aware streaming API、sample rate、`preprocessor.window_stride`、`encoder.subsampling_factor`、stream-local encoder cache、previous hypothesis、previous decoder output を明示的に扱います。

`backend.chunk_size_samples` は raw audio sample 数の contract であり、NeMo encoder chunk size ではありません。
worker は raw sample chunk と encoder token chunk を分け、`CacheAwareStreamingAudioBuffer` と `conformer_stream_step(...)` を使って cache / hypothesis / decoder output を更新します。
`audio_accepted` は audio chunk が stream state に入ったことだけを示し、partial / final transcript 成功とは別です。

#### finite attention context

現行 worker policy では、streaming backend として成立するには finite attention context が必要です。
restore 直後の context が full context `[-1, -1]` の場合、worker は supported attention contexts を取得し、現行固定 policy の `[70, 1]` が supported contexts に含まれる場合だけ `set_default_att_context_size([70, 1])` を呼びます。

NeMo の `set_default_att_context_size(...)` は unsupported look-ahead の場合に warning を出した後も context 設定と `setup_streaming_params()` を続け得ます。
そのため FluentAudio worker は NeMo warning を成功扱いせず、呼び出し前に supported contexts を検査します。
supported contexts が `[[-1, -1]]` だけなら fail closed します。

#### full-context / offline / simulated streaming

full-context / offline ASR は、発話全体または file 全体を処理して complete transcript を返す policy です。
NeMo docs には full-context model を simulated streaming に使う道がありますが、accuracy / latency tradeoff を持つ別 policy です。

現行 `nemo_rnnt_streaming` backend は streaming 専用であり、`TranscribeAudio` service の non-streaming transcription backend ではありません。
full-context model を暗黙に simulated streaming として受け入れません。
これを使う場合は、`nemo_rnnt_streaming` の成功条件を弱めるのではなく、別 backend policy として設計します。
`nemo_offline_transcribe` はこの別 backend policy です。
working tree 上では backend / worker / unit tests が作成済みで、PO 検証により real worker health、raw `FLOAT32LE` fixture transcription、opt-in file-source full ROS graph smoke は通過済みです。
accuracy 評価、`TranscribeAudio` service integration、NIM / Riva / gRPC backend readiness、generic live microphone ASR は未検証です。

#### language profile

NIM support matrix の language profile、local `.nemo` の vocabulary / metadata、FluentAudio の `backend.language` は同一 contract ではありません。
NIM の `type=default` は language auto-detect と language code output、`type=prompt` は client language code を要求する profile、`type=indic` は Indic 最適化 profile です。
`ja-JP` は NIM support matrix では default / prompt の対象であり、indic の対象ではありません。

local worker は model config が `languages` または `language` metadata を公開する場合だけ `backend.language` と exact match で照合します。
公開されない場合、language だけを理由に拒否しません。
その場合でも vocabulary に `<ja-JP>` があることは日本語を扱う前提と矛盾しない根拠に留まり、日本語 speech fixture の transcription 成功を代替しません。
現行 SO101 voice frontend profile は `backend.language: en` なので、日本語 Parakeet 検証には未適合です。

#### worker health / trace

worker `health` は startup readiness の gate です。
`.nemo` restore、model class extraction、sample rate、RNNT / Transducer capability、cache-aware streaming API、language metadata、finite attention context、streaming params setup を audio 到着前に検証します。
失敗する backend を runtime audio arrival まで遅延させません。

公開 JSONL `health_ok` response と opt-in trace / stderr は別の情報面です。
`health_ok` response は model class、cache-aware streaming flag、sample rate、channels、audio encoding、streaming flag、partial capability、language、chunk size、partial interval を返します。
selected finite attention context と effective `streaming_cfg` は公開 response field ではなく、worker internal validation と trace / stderr の証跡として扱います。

#### full ROS graph validation

full ROS graph validation は backend 単体の成功とは別の代表検証です。
検証対象は `fa_in -> fa_sample_format -> fa_resample -> fa_vad -> fa_asr` のような実際の ASR-ready stream 経路で、`AudioFrame.source_id` / `stream_id`、sample rate、encoding、channels、timeline continuity、VAD open / close、ASR event、non-empty final result を合わせて確認します。

### NeMo cache-aware streaming の必須 contract

Parakeet RNNT の cache-aware streaming は、restore 直後の model に raw audio を渡すだけでは成立しません。
NIM / Riva serving stack を使わない local `.nemo` backend では、worker が次の contract を自分で満たす必要があります。

1. model class が RNNT / Transducer 系であることを確認する。
2. encoder が cache-aware streaming API を持つことを確認する。
3. model sample rate が backend config と一致することを確認する。
4. supported attention contexts を取得し、finite attention context が成立することを確認する。
5. `preprocessor.window_stride` と `encoder.subsampling_factor` を取得する。
6. raw sample chunk と encoder token chunk を混同せず、streaming params を設定する。
7. stream start ごとに encoder cache、previous hypothesis、previous decoder output を初期化する。
8. step ごとに cache 長込みの maximum audio / positional encoding length を成立させる。
9. `conformer_stream_step(...)` の戻り値から cache / hypothesis を更新する。
10. partial / final text を transcript 成功と `audio_accepted` から分けて扱う。

NeMo の `set_default_att_context_size(...)` は、unsupported look-ahead の場合に warning を出します。
しかし warning 後も `att_context_size` をセットし、`setup_streaming_params()` を呼びます。
そのため FluentAudio worker は NeMo warning を成功扱いせず、呼び出し前に supported contexts を検査します。
supported contexts が full context `[-1, -1]` だけなら fail closed します。

### NeMo 2.7.3 source に基づく step contract

local runtime の NeMo 2.7.3 では、cache-aware streaming は次の API 群の contract で成立します。

| API | backend が守ること |
| --- | --- |
| `ConformerEncoder.setup_streaming_params(...)` | `chunk_size`、`shift_size`、`cache_drop_size`、`last_channel_cache_size`、`valid_out_len`、`pre_encode_cache_size`、`drop_extra_pre_encoded` を明示的に確立する。 |
| `ConformerEncoder.get_initial_cache_state(...)` | `start` ごとに stream-local cache を作り、session を跨いで共有しない。 |
| `CacheAwareStreamingAudioBuffer` | raw audio を NeMo が要求する chunk / length に分割する。audio sample 数と encoder feature length を同じ単位として扱わない。 |
| `ConformerEncoder.set_max_audio_length(...)` | cache を含む sequence length に対して positional encoding を不足させない。API が無い場合は fail closed。 |
| `conformer_stream_step(...)` | encoder cache、previous hypotheses、previous decoder output を受け取り、次 step 用の状態へ更新する。 |

`backend.chunk_size_samples` は NeMo encoder chunk size ではありません。
現行 worker は raw sample と encoder token の単位を次の計算で橋渡しします。

```text
encoder_token_samples = sample_rate_hz * preprocessor.window_stride * encoder.subsampling_factor
requested_encoder_chunk_size = ceil(chunk_size_samples / encoder_token_samples)
effective_encoder_chunk_size = max(10, requested_encoder_chunk_size)
shift_size = max(1, effective_encoder_chunk_size // 2)
left_chunks = 2
```

この計算は model setup の完了を意味しません。
attention context が unsupported なら、この計算へ進む前に fail closed します。

### `nemo_rnnt_streaming` の phase 別アルゴリズム

`nemo_rnnt_streaming_worker` は次の phase を順に成立させます。

1. model restore
   - `.nemo` を restore し、NeMo log を stdout JSONL protocol から隔離する。
   - restore 失敗、model object 不成立なら fail closed。
2. config extraction
   - sample rate、window stride、subsampling factor、attention context、supported contexts、language token、decoder strategy を取得する。
   - 不足値を default で埋めない。
3. capability validation
   - RNNT / Transducer、cache-aware API、sample rate、mono/float32 contract を確認する。
   - capability が説明不能なら `health_ok` を返さない。
   - model config が `languages` または `language` を公開する場合だけ `backend.language` と照合する。公開されない場合は language で拒否せず、speech fixture による日本語 transcript 検証へ回す。
4. attention context validation
   - restore state が full context `[-1, -1]` なら supported contexts を確認する。
   - requested finite context `[70, 1]` が supported contexts にある場合だけ設定する。
   - supported contexts が `[[-1, -1]]` だけなら fail closed。
5. streaming params setup
   - raw sample chunk と encoder token chunk を橋渡しし、`setup_streaming_params(...)` を呼ぶ。
   - effective `streaming_cfg` を trace できる形で保持する。
6. start
   - start command に含まれる config で stream-local buffer、encoder cache、previous hypotheses、previous decoder output、accepted sample count を初期化する。
   - active session がすでにある場合は拒否する。
   - `start` が prior `health` なしで届いた場合は、start message の config から restore / capability validation / streaming setup を行う。
   - `health` 後に `start` が届いた場合は、start message の config を health 済み worker config と比較し、不一致なら `start stream contract does not match worker health contract` として fail closed する。
   - backend wrapper は health と start に同じ resolved config を送る。
7. audio validation
   - `session_id`、wire encoding、byte length、sample count、finite / normalized range を検証する。
   - PCM16、WAV、stereo、別 sample rate を変換して継続しない。
8. buffer drain
   - validated samples を buffer に append する。
   - buffer が返す推論可能 chunk だけを step へ渡す。
   - chunk が無い場合は transcript なしの accepted state とする。
9. streaming step
   - `last_channel_cache_size + chunk_time_length` を required maximum length として計算する。
   - `set_max_audio_length(...)` で positional encoding length を成立させる。
   - `conformer_stream_step(...)` を呼び、cache / hypothesis / decoder output を更新する。
   - tensor shape mismatch を empty transcript にせず worker error とする。
10. partial / drain / finish
   - non-empty text かつ `emit_partial=true` の場合だけ partial を返す。
   - `drain` は新規 audio を追加せず、現在の partial state だけを返す。
   - `finish` は accepted audio が 1 sample 以上あることを確認し、残り chunk を処理して final を返す。
   - final text が空でも prior non-empty partial がある場合は、backend wrapper が latest partial を final commit 候補にできます。
   - final も prior partial も空の場合は transport completion であり、speech window の ASR 成功として扱わない。
11. error reporting / trace
   - `FLUENT_AUDIO_NEMO_RNNT_WORKER_TRACE_FILE` / `FLUENT_AUDIO_NEMO_RNNT_WORKER_TRACE_STDERR` で command、buffer、step、exception、final generation を記録する。
   - trace は fallback ではなく、失敗 phase を追うための証跡である。

公開 JSONL `health_ok` response と worker trace は同じ情報面ではありません。
現行 health response keys は model class、cache-aware streaming flag、sample rate、channels、audio encoding、streaming flag、final/partial capability、language、chunk size、partial interval です。
selected finite attention context と effective `streaming_cfg` は health response keys には含まれません。
それらは `health_ok` 返却前に worker 内部で検査 / 設定されますが、`fa_asr` が health response field として検証しているわけではありません。
selected context と effective `streaming_cfg` は opt-in trace、stderr、worker internal validation の証跡として扱い、公開 JSONL health contract とは分けます。

### full ROS graph / file source の ASR-ready stream contract

file source で full ROS graph を検証する場合、`fa_in` は `pcm_file_reader` backend として headerless raw PCM だけを読みます。
WAV / OGG / MP3 / FLAC decode、resample、downmix、gain、normalization は `fa_in` では行いません。
file の形式は `audio.sample_rate`、`audio.channels`、`audio.bit_depth`、`audio.encoding`、`audio.layout` で明示し、file path から推測しません。
有限 file source では `startup.required_subscribers >= 1` により、downstream subscriber が matched してから source read を開始します。
これは frame 再送や loop fallback ではなく、最初の finite frame を ROS discovery 前に捨てないための source-read gating です。
`fa_sample_format` / `fa_resample` 後に ASR-ready stream が変わる場合、`fa_vad` と `fa_asr.expected_stream_id` は raw file stream ではなく、16 kHz mono `FLOAT32LE` の最終 stream identity に合わせます。

現行 SO101 voice frontend profile の full pipeline binding は次です。

```text
raw mic
  -> fa_sample_format
  -> fa_resample16k
  -> fa_dc_offset_removal
  -> fa_high_pass
  -> fa_vad / fa_asr / fa_kws / fa_turn_detector
```

ASR-ready stream は ROS topic `audio/high_pass/frame`、`AudioFrame.stream_id=audio/high_pass/mic` です。
`fa_asr.expected_source_id=mic`、`fa_asr.expected_stream_id=audio/high_pass/mic`、`control.speech_control.source_id=mic`、`control.speech_control.stream_id=audio/high_pass/mic` を profile contract として揃えます。
現行 code は control config と `expected_*` の一致を起動時に強制していないため、この一致は profile 側の binding contract です。

archive branch は `audio/archive_pcm16/frame` / `audio/archive_pcm16/mic` として `fa_audio_window` 用に分岐します。
これは PCM16 archive/export window であり、ASR backend input ではありません。

`TranscribeAudio` service も同じ ASR-ready timeline と backend contract を使います。request start/end が sample boundary と一致しない場合は、`floor(delta_ns * sample_rate / 1e9)` / `ceil(delta_ns * sample_rate / 1e9)` で selected sample index を決定します。response の整数 ns range は selected start index を `floor(index * 1e9 / sample_rate)`、selected end index を `ceil(index * 1e9 / sample_rate)` で frame start に加算して表現し、selected samples を覆う範囲として返します。整数 ns で正確に表現できない sample 境界では、前 frame の exact end boundary を整数化した `[floor,ceil]` envelope 内の next start を contiguous として append します。next start が floor より前なら overlap として reject し、ceil より後なら append は許可しますが、slice がその区間を跨ぐと `range_not_continuous` を返します。backend には selected range の float32 samples だけを渡します。backend が返す transcript segment start/end は selected range からの相対 sample index であり、response では selected start から `floor(start_sample * 1e9 / sample_rate)` / `ceil(end_sample * 1e9 / sample_rate)` で absolute unix ns に変換します。response の `time_range` と `audio_window_ref.time_range` も selected range を返します。`fa_audio_window` が提供する PCM16 archive/export window は証拠 clip 用であり、`fa_asr` はそこから WAV decode や PCM16 -> float32 conversion を行いません。両者を同じ time range で参照する場合は、system profile で同じ upstream stream から分岐し、`AudioFrame.header.stamp` を保存することで time alignment を成立させます。
`timeline.timestamp_alignment_tolerance_ms` 内の bounded overlap / gap は sample boundary に寄せます。
tolerance 超過 overlap / gap や coverage failure は backend に空 audio や短い audio を渡さず、ASR-ready timeline error として扱います。
これは backend algorithm ではなく、backend へ渡す前の `fa_asr` timeline contract です。
`window_not_found` は ASR-ready timeline が空で requested range を判定できない状態、`range_not_continuous` は保持済み frame はあるが requested range 内に gap があり連続 coverage できない状態です。後者は「未蓄積」ではなく coverage failure なので、backend に短い audio や空 audio を渡さず、明示 error として返します。

pre-43fabab3 の file-source full graph 試行で観測された ASR timeline gap は、この contract により backend へ短い audio や空 audio を渡して成功扱いしなかった履歴証跡です。
commit `43fabab3` で `pcm_file_reader` は full chunk / partial final chunk のどちらも actual PCM frame 数から media-clock timestamp を生成するようになりました。
この修正後、commit `5e3ae5a3` の opt-in smoke は local Parakeet `.nemo` + `nemo_offline_transcribe` 経路で file-source full ROS graph の expected text `天 気` を含む final ASR result と、expected text 指定なしの non-empty final ASR result を確認しています。
そのため、旧 timeline gap を現行 file-source full graph の未完了理由として記載しません。
