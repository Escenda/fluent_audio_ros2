# fluent_audio_system テスト設計

## Unit Test

- `FA-SYS-SPEC-001`: missing config path で `RuntimeError`。
- `FA-SYS-SPEC-002`: YAML root が mapping でなければ `RuntimeError`。
- `FA-SYS-SPEC-003`: enabled node の `params_file` が空なら `RuntimeError`。inline `parameters` だけでは node を起動しない。
- `FA-SYS-SPEC-004`: missing params file で `RuntimeError`。
- `FA-SYS-SPEC-005`: remappings schema を検証する。
- `FA-SYS-SPEC-006`: `${share:...}` / `${env:...}` path を展開する。
- `FA-SYS-SPEC-007`: `fa_in_enabled` / `fa_out_enabled` が `true` / `false` 以外なら `RuntimeError`。
- `FA-SYS-SPEC-008`: site binding override は `fa_in` / `fa_out` の source/sink id と、source-bound audio AI node の `expected_source_id` だけを parameters に追加する。`fa_in_source_id` は `audio.device_selector.mode=id` として渡す。
- `FA-SYS-SPEC-009`: system config は Pydantic v2 model で境界検証する。
- `FA-SYS-SPEC-010`: `fa_in_enabled=true` かつ `fa_in_source_id` 空なら `RuntimeError`。
- `FA-SYS-SPEC-011`: `fa_out_enabled=true` かつ `fa_out_sink_id` 空なら `RuntimeError`。
- `FA-SYS-SPEC-012`: `remappings` は mapping 形式だけを受け付け、pair/list/object 形式を受け付けない。
- `FA-SYS-SPEC-013`: `analysis` group に AI package が含まれる場合は `RuntimeError`。
- `FA-SYS-SPEC-014`: system group category と package category が一致しない場合は `RuntimeError`。
- `FA-SYS-SPEC-015`: SO101 microphone frontend profile が `fa_in -> fa_sample_format -> fa_resample` の明示 format pipeline を展開する。
- `FA-SYS-SPEC-016`: SO101 TTS output profile が `fa_tts -> fa_resample -> fa_sample_format -> fa_mix -> fa_out` の明示 playback pipeline を展開する。
- `FA-SYS-SPEC-017`: `${env:...}` path を展開し、未設定または空なら `RuntimeError`。
- `FA-SYS-SPEC-018`: required package list は `fa_interfaces` / `fluent_audio_system` と enabled node package だけを出力し、disabled node と重複 package を含めない。
- `FA-SYS-SPEC-019`: required package list は pre-build resolver として node `params_file` の実体や node package share directory を要求しない。ただし enabled node の `params_file` 欠落は schema error として失敗する。
- `FA-SYS-SPEC-020`: `io` group は source / sink / utility IO packages を IO category として受け付ける。
- `FA-SYS-SPEC-021`: voice frontend profile では `fa_vad.input_topic`、`fa_kws.audio_topic`、`fa_turn_detector.audio_topic`、`fa_asr.audio_topic` / `fa_asr.expected_stream_id`、`fa_audio_embedding.input_topic` / `fa_audio_embedding.expected_stream_id` が同じ stream identity を指す。
- `FA-SYS-SPEC-022`: generated system config で `fa_in(backend.name=pcm_file_reader) -> fa_encode -> fa_decode -> fa_sample_format -> fa_gain -> fa_out(backend.name=pcm_file_writer)` の明示 codec + processing pipeline を接続し、source/sink adapter ではなく codec / processing node で encode / decode / format conversion / gain stage が行われることを検証する。
- `FA-SYS-SPEC-023`: `run.py` / `fluent_audio_system.launch.py` は launch argument default を持たず、enabled node の `node_name` 欠落は `RuntimeError`。
- `FA-SYS-SPEC-024`: package 別 parameter contract により、`params_file` と inline `parameters` の ROS topic / audio stream identity 同値使用を `RuntimeError` として拒否する。
- `FA-SYS-SPEC-025`: VLAbor include action fixture は `type: include` / `package: fluent_audio_system` / `launch: run.py` を固定し、`args` を `config` と site binding だけに限定する。
- `FA-SYS-SPEC-026`: enabled node の `params_file` が空 file または別 node 用 params file の場合は `RuntimeError`。
- `FA-SYS-SPEC-027`: runtime backend package の effective `backend.name` が未指定または空なら `RuntimeError`。

## Integration Test

- sample config から `fa_in` node action が生成される。
- disabled `fa_out` は生成されない。
- remappings が launch node に渡される。
- generated system config で `fa_out(backend.name=pcm_file_writer) -> fa_in(backend.name=pcm_file_reader)` の順に raw PCM file sink/source node を展開し、同一 topic で接続できることを検証する。
- generated system config で `fa_out(backend.name=network_pcm_sender) -> fa_in(backend.name=network_pcm_receiver)` の順に raw PCM UDP sink/source node を展開し、同一 topic で接続できることを検証する。
- generated system config で `fa_in(backend.name=pcm_file_reader) -> fa_sample_format -> fa_gain -> fa_out(backend.name=pcm_file_writer)` の明示 processing pipeline を接続し、source/sink adapter ではなく processing node で format conversion / gain が行われることを検証する。
- generated system config で `fa_in(backend.name=pcm_file_reader) -> fa_encode -> fa_decode -> fa_sample_format -> fa_gain -> fa_out(backend.name=pcm_file_writer)` の明示 codec + processing pipeline を接続し、source/sink adapter ではなく processing node で codec / format conversion / gain stage が行われることを検証する。
- Docker / ROS2 install 環境で `python3 -m pytest src/system/fluent_audio_system/test` を実行し、unit / launch / integration / E2E test を収集する。
- fixture YAML を `load_system_config()` で読み、`${share:fluent_audio_system}` 展開、missing params file、invalid schema を検証する。
- unit test で `${env:...}` が `config`、`params_file`、inline `parameters` の string / string list 値として展開されることを検証する。
- package-owned SO101 profile config を `load_system_config()` で読み、default enabled node が `fa_in` / `fa_out` だけであることを検証する。
- package-owned SO101 microphone frontend profile config を `load_system_config()` で読み、`fa_in -> fa_sample_format -> fa_resample` の順で enabled node が展開されることを検証する。
- package-owned SO101 TTS output profile config を `load_system_config()` で読み、`fa_tts -> fa_resample -> fa_sample_format -> fa_mix -> fa_out` の logical pipeline が展開され、TTS の `FLOAT32LE/32` が playback 前に 48kHz `PCM16LE/16` へ明示変換されること、OpenJTalk dictionary path が `${env:FLUENT_AUDIO_OPENJTALK_DICT_DIR}` から明示展開されることを検証する。
- package-owned SO101 TTS output profile config は `FLUENT_AUDIO_OPENJTALK_DICT_DIR` 未設定時に起動前 config 展開で `RuntimeError` になることを検証する。
- package-owned sample config が default enabled node を `fa_in` / `fa_out` / `fa_sample_format` / `fa_resample` にし、`PCM16LE` を `fa_resample` へ直結しないことを検証する。disabled codec sample としては `fa_encode` / `fa_decode` が明示 external command backend と topic contract を持つことを検証する。disabled TTS playback sample としては `fa_tts -> fa_resample_tts -> fa_sample_format_tts -> fa_mix` の明示変換 parameters を持つことを検証する。
- package-local `setup.cfg` により壊れた ambient pytest plugin を読まずに実行する。
- required package CLI が `--config` から system config を読み、1 行 1 package で base package と enabled node package を出力することを検証する。
- required package CLI が missing config / invalid schema / enabled node の `params_file` 欠落を成功扱いしないことを検証する。
- required package CLI が source-tree の package-owned profile から、未 build package の `${share:...}` params を解決せず enabled package list を出せることを検証する。
- `io` group が `fa_in` / `fa_out` / `fa_record` / `fa_stream` を含む IO package を受け付けることを検証する。file source / file sink / network source / network sink は package 名ではなく `fa_in` / `fa_out` backend config として検証し、standalone source/sink package は拒否する。
- sample / SO101 profile の voice frontend stream binding が `VadState.stream_id` と AI consumer の `expected_stream_id` 契約に一致することを検証する。
- package 別 parameter contract が `fa_sample_format`、`fa_resample`、AI consumer、routing list parameter、`fa_out` の inline topic / stream identity 同値使用を拒否することを検証する。
- package 別 parameter contract が `params_file` の flat / nested `ros__parameters` に含まれる topic / stream identity 同値使用を拒否することを検証する。
- `vlabor_include_action.yaml` fixture を読み、VLAbor profile action に backend/model/secret/detail tuning を渡す key が入っていないことを検証する。

## Launch Test

- `ros2 launch fluent_audio_system run.py config:=... fa_in_enabled:=... fa_out_enabled:=... fa_in_source_id:=... fa_out_sink_id:=...` が system launcher を include する。
- VLAbor profile action fixture が `fluent_vision_system/run.py` と同じ `type: include` 形で `fluent_audio_system/run.py` を呼ぶ。
- `ros2 launch fluent_audio_system run.py config:=... fa_in_enabled:=true fa_out_enabled:=false fa_in_source_id:=hw:... fa_out_sink_id:=disabled` が `fa_in` に raw source id override を `audio.device_selector.mode=id` で渡す。
- `fa_in_source_id` が指定された場合に `fa_vad` / `fa_kws` / `fa_asr` / `fa_turn_detector` / `fa_audio_embedding` へ `expected_source_id` override を渡す。
- launch file は `Node` action に params/remappings を直接渡し、temporary YAML の生成や書き換えをしない。
- `setup.py` が `list_required_packages` console script を install する。
- `ros2 launch fluent_audio_system fluent_audio_system.launch.py config:=<generated-system-yaml>` で `fa_in(backend.name=pcm_file_reader) -> fa_out(backend.name=pcm_file_writer)` を接続し、raw PCM bytes が同一 payload として file sink に到達することを検証する。
- `ros2 launch fluent_audio_system run.py config:=<generated-system-yaml>` で同じ file source / file sink 経路を実行し、VLAbor が include する entrypoint 経由でも raw PCM bytes が到達することを検証する。
- `ros2 launch fluent_audio_system fluent_audio_system.launch.py config:=<generated-system-yaml>` で `fa_in(backend.name=network_pcm_receiver) -> fa_out(backend.name=network_pcm_sender)` を接続し、raw PCM UDP packet が同一 payload として network sink に到達することを検証する。
- `ros2 launch fluent_audio_system fluent_audio_system.launch.py config:=<generated-system-yaml>` で `fa_in(backend.name=pcm_file_reader) -> fa_sample_format -> fa_gain -> fa_out(backend.name=pcm_file_writer)` を接続し、PCM16 入力が `FLOAT32LE/32` へ明示変換され、gain 後の raw float32 bytes が file sink に到達することを検証する。
- `ros2 launch fluent_audio_system fluent_audio_system.launch.py config:=<generated-system-yaml>` で `fa_in(backend.name=pcm_file_reader) -> fa_encode -> fa_decode -> fa_sample_format -> fa_gain -> fa_out(backend.name=pcm_file_writer)` を接続し、external codec backend と明示 processing stage を経由した raw float32 bytes が file sink に到達することを検証する。
