# fluent_audio_system 仕様書

## 責務

FluentAudio の system config を読み、enabled group / node を ROS2 launch action として展開します。
同じ config から、VLAbor / Docker build 側が必要な ROS2 package を解決するための package list も生成します。

## 非責務

- backend/model の自動選択
- backend/model/format capability の推測補完
- device 自動推測
- params file の生成・書き換え
- missing params file の fallback
- VLAbor profile 形式の解釈

## Input

| 種別 | 名前 | 内容 |
| --- | --- | --- |
| launch argument | `config` | FluentAudio system YAML の絶対パス、またはカンマ区切りの明示 config path list。複数指定時は左から右の順に composition する |
| launch argument | `fa_in_enabled` | profile で宣言された `fa_in` を site profile 側で有効化するか。`true` / `false` のみ。launch default は持たない |
| launch argument | `fa_out_enabled` | profile で宣言された `fa_out` を site profile 側で有効化するか。`true` / `false` のみ。launch default は持たない |
| launch argument | `fa_in_source_id` | profile 由来の raw ALSA capture source id |
| launch argument | `fa_out_sink_id` | profile 由来の raw ALSA playback sink id |
| CLI argument | `--config` | package list 生成対象の FluentAudio system YAML、またはカンマ区切りの明示 config path list。複数指定時は左から右の順に composition する |

## Config Schema

YAML は読み込み境界で Pydantic v2 model に変換し、以降の launch 展開では型確定済みの config として扱います。未検証 dict を深い層へ流しません。

```yaml
system:
  default_start_delay: 0.2
  inter_group_delay: 0.5

groups:
  - id: io
    enable: true
    nodes:
      - id: fa_in
        enable: true
        package: fa_in
        exec: fa_in_node
        node_name: fa_in
        namespace: ""
        params_file: /path/to/fa_in.yaml
        parameters: {}
        remappings:
          audio/frame: robot/audio/frame
```

## Config Composition

`fluent_audio_system` の `config` launch argument と `list_required_packages --config` は、単一 system YAML に加えて、カンマ区切りの明示 config path list を受け付けます。空白や暗黙探索ではなく、呼び出し側が渡した path を左から右の順に読み、group / node の展開順と required package list の順序もその順序を保持します。

composition は child-side FluentAudio package 内の config composition です。`so101_voice_frontend.yaml,so101_agent_audio_tools.yaml` のように package-owned system config を 1 つの FluentAudio system config として読むための機能です。package-local config / CLI composition と、実 SO101 FluentAudio profile pair の runtime smoke は child repo 内で検証済みですが、親 repo の VLAbor profile integration、Agent Runtime / MCP client integration、実 R2/S3 client upload、World Station evidence linkage、実 SO101 device / model provisioning が検証済みであることを意味しません。multi-config の代表 runtime launch smoke は `FA-SYS-SPEC-036`、実 SO101 FluentAudio profile pair smoke は `FA-SYS-SPEC-039` で扱います。

複数 config を compose する場合、各 file の `system.default_start_delay` と `system.inter_group_delay` は一致している必要があります。これらは compose 後の launch timing を 1 つの system として扱うための system-level contract であり、file ごとに異なる timing を混在させません。

group id は compose 対象全体で重複不可です。enabled node id も compose 対象全体で重複不可です。disabled node id は launch 対象外ですが、enabled node の identity 衝突を隠すための互換 layer は持ちません。

## Fail Closed

- `config` が空
- `config` / `--config` のカンマ区切り list に空 segment が含まれる
- `config` が存在しない
- YAML が不正で parse できない
- YAML root が mapping ではない
- `system` が存在しない、または mapping ではない
- `system.default_start_delay` / `system.inter_group_delay` が未指定
- `system.default_start_delay` / `system.inter_group_delay` が負数
- compose 対象 file 間で `system.default_start_delay` / `system.inter_group_delay` が一致しない
- compose 対象 file 間で group id が重複する
- compose 対象 file 間で enabled node id が重複する
- `groups` が存在しない
- `groups` が sequence ではない
- enabled / disabled に関わらず group の `enable` が未指定
- enabled / disabled に関わらず node の `enable` が未指定
- enabled group に `nodes` が存在しない
- enabled node の `id`, `package`, `exec` が空
- enabled node の `node_name` が空
- enabled node の `params_file` が空
- enabled node の `params_file` に `node_name` / `package` / `/**` のいずれの `ros__parameters` block も存在しない
- `analysis` group に VAD / KWS / ASR / Turn Detector / audio embedding などの buildable AI package が含まれる
- `package.xml` を持たない roadmap placeholder package が system config の `package` に指定されている
- system group category と package category が一致しない
- enabled node の `params_file` が存在しない
- `remappings` の from / to が空
- `remappings` が mapping ではない
- inline `parameters` が ROS parameter として扱えない型
- package 別 parameter contract で ROS topic と audio stream identity に同じ値を使っている
- package 別 parameter contract の topic / stream identity key に string または string list 以外の値を渡している
- runtime backend package の effective `backend.name` が未指定または空。対象は `fa_in` / `fa_out` / codec backend、correction backend、deterministic analysis backend、AI backend、TTS backend
- `${env:VAR_NAME}` 参照先の環境変数が未設定または空
- `fa_in_enabled` / `fa_out_enabled` が `true` / `false` 以外
- `fa_in_enabled=true` なのに `fa_in_source_id` が空
- `fa_out_enabled=true` なのに `fa_out_sink_id` が空

`fluent_audio_system` は `backend.name`、site binding、path 展開、topic / stream identity など、system YAML から判定できる contract を起動前に検証する。個別 backend の sample format、sample rate、model provider、payload encoding、model IO schema などの capability は各 node/backend の仕様に属する。system はそれらを推測して埋めず、各 node が startup failure / frame rejection / runtime fatal / explicit error result として fail closed できる設定だけを渡す。

## Path Expansion

`config`、`params_file`、inline `parameters` の string / string list 値は `${share:package_name}` と `${env:VAR_NAME}` を展開できます。package が見つからない場合、または env が未設定 / 空の場合は起動失敗します。これは明示された path の展開であり、missing model / missing params_file の fallback ではありません。

`parameters` は `params_file` に対する明示 override としてのみ扱います。inline `parameters` だけで node を起動することは許可しません。

`params_file` は存在するだけでは不十分です。enabled node の `node_name`、`package`、または `/**` root のいずれかに `ros__parameters` block が必要です。空 file や別 node 用 params file の流用は、runtime default へ落ちる fallback として扱い、起動前に失敗させます。

## Topic / Stream Identity Contract

`input_topic` / `output_topic` / `audio_topic` / `vad_topic` / `input_topics` / `output_topics` などの ROS transport topic と、`input_stream_id` / `expected_stream_id` / `expected.stream_id` / `output.stream_id` / `input_stream_ids` などの `AudioFrame.stream_id` identity は別 contract です。

`fluent_audio_system` は package ごとの明示 parameter contract で `params_file` の `ros__parameters` と inline `parameters` を検証し、同じ値を topic と stream identity の両方に使う config を起動前に拒否します。`params_file` は node name、package name、`/**` の順に既知 ROS params block を読み、nested YAML は dotted parameter key に flatten して検証します。先頭 `/` の有無だけが違う値も同一 identity として扱います。suffix だけで任意 key を推測せず、supported package の既知 parameter key だけを検証対象にします。

## Required Package List

`ros2 run fluent_audio_system list_required_packages --config <system.yaml>` は、system config を build 解決用の validation で読み、build に必要な package を 1 行 1 package で出力します。

出力順は固定です。

1. `fa_interfaces`
2. `fluent_audio_system`
3. system config の enabled node package を group / node の展開順で重複排除したもの

disabled group / disabled node の package は出力しません。`config` 欠落、invalid schema、enabled node の `params_file` 欠落、category mismatch は CLI でも失敗扱いにし、部分的な package list は成功として返しません。

この CLI は pre-build package resolver なので、enabled node の `params_file` 実体確認、node params 内の `${share:...}` 解決、node params 内の `${env:...}` 解決は行いません。まだ build していない package の share directory が存在しない段階でも、system config から enabled package set を決定できる必要があります。launch 実行時は `load_system_config()` が `params_file` 実在と parameter path 展開を fail closed で検証します。

## Package-owned Profiles

`config/profiles/so101.yaml` は SO101 用の標準 profile であり、default enabled node は `fa_in` / `fa_out` に限定する。format pipeline は定義するが default では無効にし、site / launch 側が明示的に有効化する。

`config/profiles/so101_mic_frontend.yaml` は microphone frontend の明示 pipeline である。`fa_in` の default `PCM16LE/16/48kHz` stream を `fa_sample_format` に渡し、`FLOAT32LE/32/interleaved` に変換してから `fa_resample` へ渡す。`fa_resample` に PCM16 を直結しない。さらに `fa_dc_offset_removal` と `fa_high_pass` を dedicated processing node として挟み、DC offset removal と低域ノイズ抑制を `fa_in` に隠さない。

`config/profiles/so101_kws_frontend.yaml` は microphone frontend に VAD / KWS を接続する明示 pipeline である。`fa_in -> fa_sample_format -> fa_resample -> fa_dc_offset_removal -> fa_high_pass -> fa_vad / fa_kws` を展開し、`fa_vad` の Silero worker と `fa_kws` の sherpa-onnx worker は system config の inline `parameters` に閉じる。worker command、model path、provider、keywords file は `${env:...}` で明示し、未設定なら system config 展開時に起動失敗する。VLAbor profile / `run.py` launch arguments には KWS backend/model 詳細を渡さない。

`config/profiles/so101_voice_frontend.yaml` は microphone frontend に VAD / KWS / ASR / Turn Detector、`fa_dialogue`、archive branch を接続する明示 pipeline である。主処理系は `fa_in -> fa_sample_format -> fa_resample -> fa_dc_offset_removal -> fa_high_pass -> fa_vad / fa_kws / fa_asr / fa_turn_detector` を展開し、ASR は `audio/high_pass/frame` / `audio/high_pass/mic` を入力にする。ASR は `fa_audio_window` に依存せず、archive 用 PCM16 window から hidden conversion して文字起こししない。archive branch は `audio/high_pass/frame` / `audio/high_pass/mic` から `fa_archive_sample_format` (`fa_sample_format`) で `PCM16LE/16/interleaved` へ明示変換し、`audio/archive_pcm16/frame` / `audio/archive_pcm16/mic` を streaming package である `fa_audio_window` へ渡す。ASR timeline と archive/export window は同じ high-pass stream から分岐し、`AudioFrame.header.stamp` を保つことで同じ time range を共有する。ASR profile は `timeline.retention_sec`、`timeline.clock`、`timeline.window_id`、`timeline.window_epoch` を system config 側に明示し、`TranscribeAudio` の `AudioWindowRef` が default identity に依存しないようにする。`fa_audio_window` は `export_audio_window` と `archive_audio_window` service を提供する。ASR は `parakeet_multilingual_buffered` backend に local multilingual Parakeet `.nemo` model path、`language_policy=auto_detect`、rolling-buffer decode cadence を渡し、Turn Detector は `smart_turn_onnx` worker の contract を system config の inline `parameters` に閉じる。`fa_dialogue` は `voice/wake_word`、`voice/asr/result`、`voice/turn_end` を受け、`conversation/turn_context` を publish する。この profile は LLM reasoning、TTS、safety policy、robot command proposal を含まない。

`config/fluent_audio_system.sample.yaml`、`config/profiles/so101.yaml`、`config/profiles/so101_mic_frontend.yaml` に含まれる VAD / KWS / ASR / Turn Detector の disabled node template も、backend name、worker command、model path / model files、provider、worker args、workspace、QoS を system config 側に持つ。KWS / ASR / Turn Detector は startup health check 用の `backend.health_args` も system config 側に持つ。disabled template であっても、enable した瞬間に backend/model 詳細を VLAbor profile や launch argument から補完する設計にはしない。

`config/profiles/so101_tts_output.yaml` は TTS playback の明示 pipeline である。`fa_tts` は `audio/tts/frame` に `FLOAT32LE/32` を publish し、`fa_resample` が `audio/tts/48k_float32` へ sample rate を明示変換する。`fa_sample_format` が `PCM16LE/16` の `audio/tts/pcm16` へ変換し、その後 `fa_mix` が `audio/output/frame` へ routing し、`fa_out` が sink adapter として再生する。`fa_out` は resample、format conversion、gain、normalize を行わない。OpenJTalk dictionary path は `backend.openjtalk_dict_dir: ${env:FLUENT_AUDIO_OPENJTALK_DICT_DIR}` として明示し、未設定時は system config 展開で起動失敗する。

## Site Binding Override

VLAbor profile から渡す site 依存値は `fa_in_enabled`、`fa_out_enabled`、`fa_in_source_id`、`fa_out_sink_id` に限定します。これらは `fa_in` / `fa_out` の起動可否と raw ALSA id だけを上書きします。`fa_in_source_id` は `fa_in` の `audio.device_selector.mode=id` / `audio.device_selector.identifier=<fa_in_source_id>` として渡し、表示名解決を混ぜません。

`config`、`fa_in_enabled`、`fa_out_enabled`、`fa_in_source_id`、`fa_out_sink_id` はすべて明示 launch argument とし、launch file には default を置かない。system config で enabled な `fa_in` / `fa_out` を site binding default で黙って抑制しない。standalone debug で IO を起動しない場合は `fa_in_enabled:=false` / `fa_out_enabled:=false` を明示する。

`fa_in_enabled=true` の場合は `fa_in_source_id` を必須とし、`fa_out_enabled=true` の場合は `fa_out_sink_id` を必須とします。空の場合に system config 内の default device へ流すことは、site binding 欠落を隠す device fallback として扱い、起動失敗にします。

`fa_in_source_id` が指定された場合、source-bound audio AI node である `fa_vad` / `fa_kws` / `fa_asr` / `fa_turn_detector` / `fa_audio_embedding` には `expected_source_id` として同じ値を渡します。これは source identity の binding であり、backend/model/threshold を profile から渡す経路ではありません。

VAD frontend を同一 stream として構成する場合、system config 側で `fa_vad.input_stream_id`、`fa_kws.expected_stream_id`、`fa_turn_detector.expected_stream_id`、`fa_asr.expected_stream_id`、`fa_audio_embedding.expected_stream_id` を同じ `AudioFrame.stream_id` に揃えます。`fa_vad.input_topic`、`fa_kws.audio_topic`、`fa_turn_detector.audio_topic`、`fa_asr.audio_topic`、`fa_audio_embedding.input_topic` は ROS transport topic であり、frame identity ではありません。`fa_in_source_id` は source identity の site binding であり、stream identity は processing config で明示します。

backend/model 名、VAD / KWS / ASR / Turn Detector / audio embedding の model path、threshold、prompt、timeout、embedding dimension、external command は system config または各 node config に閉じます。VLAbor profile や `run.py` launch arguments から backend/model 詳細を渡しません。

## VLAbor Include Action Contract

VLAbor profile からの起動 action は Fluent Vision と同じ `type: include` 形式にします。`package` は `fluent_audio_system`、`launch` は `run.py` とし、`args` は `config` と site binding だけに限定します。

```yaml
- type: include
  package: fluent_audio_system
  launch: run.py
  args:
    config: ${fluent_audio_config}
    fa_in_enabled: ${fluent_audio_fa_in_enabled}
    fa_out_enabled: ${fluent_audio_fa_out_enabled}
    fa_in_source_id: ${fluent_audio_fa_in_source_id}
    fa_out_sink_id: ${fluent_audio_fa_out_sink_id}
  enabled: ${fluent_audio_enabled}
```

`test/fixtures/vlabor_include_action.yaml` はこの profile action の canonical fixture です。VLAbor 側の profile に実際に追加するときも、この fixture と同じ key set を使います。`backend.name`、model path、threshold、prompt、endpoint、secret はこの action の `args` に追加しません。

## Combined Owner / MCP Launch Smoke

`fluent_audio_system/run.py` は、複数の system YAML から service owner node と Agent-facing adapter node を同時に展開できる必要があります。現在の代表 smoke は `test/integration/test_combined_owner_mcp_launch_smoke.py::test_fluent_audio_system_composes_owner_and_mcp_adapter_configs` です。

この smoke は test 内で生成した owner config と adapter config を `config:=owner.yaml,adapter.yaml` 形式で `ros2 launch fluent_audio_system run.py` に渡し、fake worker / generated params を使いながら、実 `fa_asr` / `fa_asr_node`、`fa_audio_window` / `fa_audio_window_node` owner services と `fa_audio_mcp` / `fa_audio_mcp_server` adapter を `fluent_audio_system` 経由で起動します。`fa_in_enabled:=false` / `fa_out_enabled:=false` の site binding を明示し、disabled site binding value は launch argument として構文的に有効な `disabled` を使います。

検証する contract は launch-managed MCP HTTP tool-call runtime behavior です。smoke は起動済み ROS graph 上で MCP adapter の ROS node が見えること、`fa_asr` が `TranscribeAudio` service owner として起動していること、`fa_audio_window` が `ExportAudioWindow` / `ArchiveAudioWindow` service owner として起動していることを待ちます。その後、MCP streamable-http `/mcp` に接続し、`list_tools` で tool surface を確認したうえで、`time_range="now-10s..now"` を使って `transcribe_audio`、`archive_audio_window`、`export_audio_window` を実 call します。adapter が ROS node clock により `20s..30s` の numeric range へ解決してから owner services を呼び、deterministic `FLOAT32LE` ASR frame と deterministic `PCM16LE` archive frame に対して、transcript / model ref / ASR window ref、WAV clip bytes、archive metadata、export result が入力 frame と resolved time range に対応することを確認し、unsupported scope が MCP error result になることも確認します。

この smoke は、`fa_asr` が transcribe owner、`fa_audio_window` が archive/export owner である境界と、child-side generated owner/adapter config を同一 `fluent_audio_system` launch graph に載せた MCP streamable-http tool-call 経路を検証します。実 `so101_voice_frontend.yaml,so101_agent_audio_tools.yaml` profile pair runtime smoke も同じ combined smoke file で扱います。親 VLAbor profile integration、親 Agent Runtime / MCP client 統合、実 R2/S3 client upload、World Station evidence linkage、実 SO101 model / device provisioning は検証済みではありません。

検証済み結果: real ASR opt-in smoke 追加後の Docker/VLAbor 代表 system validation では、SO101 profile package set を含めた isolated build が通り、`colcon test --packages-select fluent_audio_system` は `197 passed, 1 skipped`。skip は real ASR smoke の default opt-in skip であり、実 ASR model / audio fixture を与えた成功経路の検証ではない。

durable media identity fields と filesystem archive store contract は child repo 内の `fa_audio_window` / `fa_audio_mcp` package 検証で確認済みの範囲であり、この system smoke は owner / adapter launch と MCP tool-call 経路の検証である。親連携、実 R2/S3 client upload、World Station artifact creation まで完了扱いしない。

## SO101 FluentAudio Profile Pair Real ASR Opt-in Smoke

`test/integration/test_combined_owner_mcp_launch_smoke.py::test_so101_profile_pair_runs_real_asr_model_when_enabled` は、commit `c2b349f2` で追加された opt-in smoke です。default では `FLUENT_AUDIO_REAL_ASR_SMOKE=1` がない限り skip し、通常の `fluent_audio_system` test suite が実 ASR model / audio fixture を暗黙要求しないようにします。

enabled 時は、実 SO101 profile pair `${share:fluent_audio_system}/config/profiles/so101_voice_frontend.yaml,${share:fluent_audio_system}/config/profiles/so101_agent_audio_tools.yaml` を `fluent_audio_system/run.py` 経由で起動します。launch path は fake worker 版の SO101 profile pair smoke と同じです。ただし ASR model、ASR 入力 audio だけを実値に差し替え、VAD / KWS / Turn Detector は fake fixture を使います。

必須 env は `FLUENT_AUDIO_ASR_MODEL_PATH`、`FLUENT_AUDIO_REAL_ASR_AUDIO_F32_PATH` です。`FLUENT_AUDIO_ASR_MODEL_PATH` は読み取り可能 file、`FLUENT_AUDIO_REAL_ASR_AUDIO_F32_PATH` は読み取り可能かつ非空の raw `FLOAT32LE` mono 16kHz fixture でなければなりません。audio fixture は byte length が 4 の倍数で、sample は有限かつ `[-1.0, 1.0]` の normalized value だけを含む必要があります。これらが欠ける、または invalid な場合は fail closed とします。

任意 env は `FLUENT_AUDIO_REAL_ASR_EXPECTED_TEXT` と `FLUENT_AUDIO_REAL_ASR_SAMPLE_RATE` です。`FLUENT_AUDIO_REAL_ASR_EXPECTED_TEXT` が設定されている場合は transcript がその文字列を含むことを要求し、未設定の場合は transcript が非空であることを要求します。`FLUENT_AUDIO_REAL_ASR_SAMPLE_RATE` を設定する場合は `16000` だけを許可します。この smoke は resampling を隠さず、16kHz 以外を成功扱いしません。

この smoke が検証する contract は、ROS service `TranscribeAudio` が success を返すこと、segment の time range が publish した high-pass input frame に対応すること、`model_ref.backend_name` が `parakeet_multilingual_buffered`、`model_ref.backend_kind` が `asr`、`model_ref.model_path` が提供された model path と一致することです。実 SO101 device provisioning、親 VLAbor profile integration、親 Agent Runtime / MCP client、実 R2/S3 client upload、World Station evidence linkage は検証対象外です。現在の環境では実 ASR env 値が提供されていないため、実 model / audio fixture による成功経路は未検証です。
