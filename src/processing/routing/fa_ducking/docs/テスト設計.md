# fa_ducking テスト設計

## 1. 目的

`fa_ducking` の contract test は、sidechain ducking が AudioFrame 契約、設定 validation、metadata 保持、fail closed の方針を満たすことを確認する。

## 2. Unit contract test

`test/unit/test_fa_ducking_audio_frame_contract.py` で以下を検査する。

- example config が `FLOAT32LE` interleaved contract を要求すること
- launch の `config_file` が明示必須で、package default を持たないこと
- startup config validation が必須 parameter、threshold、gain、attack/release、expected format を fail closed で検証すること
- program / sidechain frame validation が stream_id、format、data size を検証すること
- node が sample loop と state mutation を backend へ委譲すること
- backend が sidechain active 判定に max age と threshold を使うこと
- backend が dB から linear に変換し、attack/release smoothing を使うこと
- 入力 sample と出力 sample の範囲違反を drop し、clamp しないこと
- output metadata は program frame を基準にし、`stream_id` を `output.stream_id` に更新すること
- package layout と colcon pytest / backend gTest integration が存在すること

## 3. Backend C++ test

- valid sidechain RMS を観測し、active sidechain で program gain が下がることを確認する。
- stale sidechain を inactive として扱うことを確認する。
- invalid sidechain / program input が state や output buffer を commit しないことを確認する。
- invalid config を constructor で拒否する。
- unknown backend status を `std::logic_error` で fail closed する。

## 4. Integration test 方針

`test/integration` では、将来 ROS2 runtime test を追加する。代表ケースは以下である。

- sidechain が threshold 未満の間は gain が release で `1.0` へ戻ること
- sidechain が threshold 以上の間は attack で duck gain へ近づくこと
- stale sidechain は inactive として扱われること
- 不正 sidechain frame は recent sidechain state を無効化すること

## 5. Launch test 方針

`test/launch` では、`config_file` 未指定で package default に fallback しないこと、明示 config で起動できること、不正 config で起動失敗することを確認する。
