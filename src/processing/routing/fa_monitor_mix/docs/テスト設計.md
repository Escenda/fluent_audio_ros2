# fa_monitor_mix テスト設計

## 1. 単体 contract test

Python の static contract test で、package-local source/config/launch が次を満たすことを確認する。

- default config が `fa_monitor_mix` key を使う
- launch default `node_name` が `fa_monitor_mix`
- `input_topics` と `input_stream_ids`、`output_topic` と `output.stream_id` を分け、topic identity と stream identity の衝突を fail closed する
- `FLOAT32LE` / `32` / `interleaved` 以外を起動時に拒否する
- 入力 frame の `source_id`、`stream_id`、format、data alignment、sample range を検証する
- missing / stale / byte length mismatch で mix を drop する
- output が `output.source_id` と `output_topic` を使う
- clamp / limiter / normalize / resample / device API を含まない
- diagnostics counter 名を publish する
- node core と `internal_monitor_mix` backend library が分かれている
- backend C++ test が input validation、gain mix、byte length mismatch、range overflow を確認する

## 2. C++ build test

親工程で `colcon build --packages-select fa_monitor_mix` を実行し、C++17 と依存関係の解決を確認する。

## 3. ROS2 smoke test

親工程で package を build 後、次を確認する。

1. default config で node が起動する。
2. `audio/program/frame` と `audio/tts/frame` に valid frame を publish すると
   `audio/monitor/frame` が publish される。
3. 片方の入力を止めると stale drop が増え、monitor frame が publish されない。
4. sample range 超過 frame を publish すると output は publish されず `range_drops` が増える。

## 4. 未検証扱いの境界

static test と py_compile だけでは ROS2 runtime の subscription/publish 動作は確認できない。
runtime smoke test まで通した場合だけ、ROS2 経路検証済みとして扱う。
