# fa_patchbay テスト設計

## 仕様 ID

| ID | 仕様 |
| --- | --- |
| FA-PATCHBAY-SPEC-001 | default config key と node default name は `fa_patchbay` |
| FA-PATCHBAY-SPEC-002 | `input_topics`、`input_stream_ids`、`output_topics`、`output_stream_ids` は同じ長さの parallel array |
| FA-PATCHBAY-SPEC-003 | route tuple は unique で、input/output topic identity と topic/stream identity は衝突しない |
| FA-PATCHBAY-SPEC-004 | expected format は `PCM16LE/16`, `PCM32LE/32`, `FLOAT32LE/32`, `interleaved` のみ |
| FA-PATCHBAY-SPEC-005 | runtime frame は source_id、configured input stream identity、format、data alignment を検証する |
| FA-PATCHBAY-SPEC-006 | valid frame は route ごとに copy され、変更 field は `stream_id` のみ |
| FA-PATCHBAY-SPEC-007 | invalid frame は drop され、counter が増える |
| FA-PATCHBAY-SPEC-008 | diagnostics は route_count、unique_input_count、input_topic.N、input_stream_id.N、resolved_input_topic.N、output_topic.N、output_stream_id.N、resolved_output_topic.N、frames_in、frames_dropped、copies_out を含む |
| FA-PATCHBAY-SPEC-009 | package layout は FluentAudio processing package 標準に従う |

## Test Case

| Test Case | 対応仕様 | 内容 |
| --- | --- | --- |
| `test_default_config_defines_explicit_patchbay_contract` | SPEC-001, SPEC-002, SPEC-003, SPEC-004 | default YAML の契約を検査する |
| `test_startup_validation_fails_closed_for_invalid_config` | SPEC-002, SPEC-003, SPEC-004 | C++ 起動時 validation が fail closed することを検査する |
| `test_runtime_frame_validation_drops_invalid_frames_before_routing` | SPEC-005, SPEC-007 | runtime validation の drop 条件を検査する |
| `test_patchbay_copies_frame_and_only_changes_stream_id` | SPEC-006 | publish 処理が `stream_id` 以外を変更しないことを検査する |
| `test_patchbay_does_not_implement_audio_processing` | SPEC-006 | DSP / format conversion / gain / mixing が実装されていないことを検査する |
| `test_diagnostics_report_routing_state_and_counters` | SPEC-008 | diagnostics key と counter を検査する |
| `test_package_layout_matches_required_processing_layout` | SPEC-009 | package layout を検査する |
| `test_colcon_runs_pytest_contracts` | SPEC-009 | colcon から pytest が実行されることを検査する |

## 未実施の検証

この package 内の unit contract test は静的検査である。ROS2 runtime での publish / subscribe 結合は親 agent 側の integration/launch test で扱う。
