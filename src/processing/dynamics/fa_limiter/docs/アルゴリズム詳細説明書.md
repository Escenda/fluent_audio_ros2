# fa_limiter アルゴリズム詳細説明書

## 入力検証

1. subscription callback で null `AudioFrame` pointer を drop する。
2. `source_id` と `stream_id` が空でないことを確認する。
3. `stream_id == input_topic` を確認する。
4. `layout == interleaved` を確認する。
5. `encoding == FLOAT32LE`、`bit_depth == 32` を確認する。
6. `sample_rate` と `channels` が config の `expected.*` と一致することを確認する。
7. `data.size()` が `channels * sizeof(float)` の倍数であることを確認する。

不一致は frame skip として扱う。別 format を暗黙変換すると format node の責務を侵食するためである。

## Hard Limiter

`internal_limiter` backend が sample 単位の hard limiter 処理を行う。node は ROS
parameter、AudioFrame metadata、diagnostics だけを扱い、sample loop を持たない。

1. validated sample bytes を 4 byte ごとに `float` として読み出す。
2. 入力 sample が finite であることを確認する。
3. `sample > threshold.linear` の場合は `threshold.linear` にする。
4. `sample < -threshold.linear` の場合は `-threshold.linear` にする。
5. それ以外の sample は変更しない。
6. 出力 bytes へ `float` として書き戻す。

`fa_limiter` の threshold limiting は limiter の明示責務である。finite sample が
`[-1.0, 1.0]` を超えていても、threshold へ制限できる場合は処理対象とする。
これは欠損値や不明形式を補う fallback ではない。sample rate、channel 数、layout、
bit depth、encoding は変更しない。

backend は non-finite input または byte alignment error では `ProcessStatus` を返し、
output buffer と `samples_limited` counter を commit しない。

## 出力

- `header`、`source_id`、sample format metadata は入力から保持する。
- `stream_id` は `output_topic` に更新する。
- diagnostics には `frames_in`、`frames_out`、`frames_dropped`、`samples_limited`、`threshold_linear` を出す。

## Node Core

`FaLimiterNode` は `rclcpp::NodeOptions` を受け取る node core として実装する。
executable entrypoint は `src/main.cpp` に分離し、graph test は node core を直接生成して
publisher / subscriber 経路を検証する。これにより launch file だけでなく ROS graph 上の
metadata 保持、`stream_id` 更新、drop 条件を実行可能な test で確認する。
