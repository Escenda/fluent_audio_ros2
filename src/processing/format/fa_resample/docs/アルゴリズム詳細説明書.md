# fa_resample アルゴリズム詳細説明書

## 責務分離

`fa_resample` は sample rate conversion だけを担当する。ROS node は parameter、QoS、mic/ref topic、stream identity、publish/drop、diagnostics、`AudioFrame` mapping を担当する。

resampler backend は ROS2 topic/message を知らない。backend は frame contract validation、sample decode/encode、finite normalized sample enforcement、stream state、resampling、metrics を担当する。

## 共通処理手順

```text
AudioFrame
  ↓
node: source_id / stream_id validation
  ↓
backend: validate FLOAT32LE/32/interleaved frame contract
  ↓
backend: decode raw FLOAT32LE bytes to interleaved float32 samples
  ↓
backend: reject non-finite or out-of-range samples
  ↓
backend: get or create stream state keyed by stream_id
  ↓
backend: reject mid-stream contract changes
  ↓
backend: resample with selected backend
  ↓
backend: encode raw FLOAT32LE bytes
  ↓
node: publish AudioFrame
```

## backend selection algorithm

起動時に `backend.name` を `BackendKind` へ変換する。

- `internal_linear_resampler`: 内部線形 backend を生成する。`backend.quality` は未設定または `debug_reference` のみ許可する。
- `speexdsp`: `backend.quality` を integer `0..10` として検証し、`libspeexdsp.so.1` を `dlopen` する。
- `soxr`: `backend.quality` を `QQ` / `LQ` / `MQ` / `HQ` / `VHQ` として検証し、`libsoxr.so.0` を `dlopen` する。

未知の backend 名、品質指定の型違い、範囲外品質、選択 backend の missing runtime dependency は起動失敗である。別 backend へ切り替えない。

`package.xml` は runtime package contract として `libspeexdsp1` と `libsoxr0` を `exec_depend` に宣言する。
実装は runtime library を `dlopen` するため、SpeexDSP / SoXR の header や link target を build dependency として要求しない。

## frame / buffer 前提

- 入力は `FLOAT32LE` / 32-bit / `interleaved` PCM。
- `stream_id` は node 側で対応する `mic.input_stream_id` / `ref.input_stream_id` と一致する。
- `sample_rate > 0`。
- `channels >= 1`。
- `data.size()` は `channels * sizeof(float)` の倍数。
- sample は finite かつ normalized range `[-1.0, 1.0]` に入っている。
- 出力 sample rate は `target_sample_rate`。
- 出力 `stream_id` は対応する `mic.output.stream_id` / `ref.output.stream_id`。

`fa_resample` は PCM16 / PCM32 を float32 に変換しない。整数 PCM を扱う場合は前段に `fa_sample_format` を明示して挟む。出力も `FLOAT32LE` / 32-bit 固定であり、PCM16 へ戻す処理や clipping 防止の clamp はここに置かない。

## stream state algorithm

backend は `stream_id` ごとに state を作る。state 作成時に次を stream contract として保存する。

- `stream_id`
- input frame contract: `encoding`、`sample_rate`、`channels`、`bit_depth`、`layout`
- `target_sample_rate`
- backend name
- backend quality

同じ `stream_id` の後続 chunk では、この stream contract と一致する場合だけ処理する。sample rate、channel count、encoding、layout、target sample rate、backend name、backend quality のどれかが変わった場合は `ProcessStatus::kStreamContractViolation` を返す。

この state は chunk ごとの処理境界をまたいで保持する。backend を chunk ごとに作り直さない。

## internal_linear_resampler

`internal_linear_resampler` は debug / reference 用の内部 backend である。quality label は `debug_reference`。

stream state は次を持つ。

- stream contract
- cumulative input frame count
- cumulative output frame count
- next output frame index
- retained input tail buffer
- retained buffer start frame index

各 output frame は cumulative output frame index から入力位置を決める。

```text
src_pos = next_output_frame_index * input_sample_rate / target_sample_rate
```

`src_pos` の `floor` と次 frame を使い、channel ごとに線形補間する。

```text
y[i] = (1 - frac) * x[floor(src_pos)] + frac * x[min(floor(src_pos)+1, last)]
```

chunk 境界で補間に必要な直前 tail frame を保持するため、frame ごとに独立した one-shot resample にはしない。

## speexdsp

`speexdsp` backend は `libspeexdsp.so.1` を runtime load し、`speex_resampler_process_interleaved_float` 相当の interleaved float path を使う。

stream state は SpeexDSP resampler state、stream contract、cumulative input/output frame count、input/output latency を持つ。stream 作成時に SpeexDSP の input/output latency API から latency を取得し、metrics に反映する。

`backend.quality` は `0..10` の integer である。範囲外または integer 以外は起動失敗にする。runtime library が無い場合も起動失敗にする。

SpeexDSP が output を遅延させ、一部 chunk で output frame 数が 0 になることは正常な streaming 挙動として扱う。ただし backend process が入力も出力も進められない場合は backend process failure とする。

## soxr

`soxr` backend は `libsoxr.so.0` を runtime load し、SoXR streaming API を使う。one-shot conversion ではなく、stream ごとに SoXR state を保持する。

quality は `QQ` / `LQ` / `MQ` / `HQ` / `VHQ` のいずれかである。選択値は SoXR quality recipe に変換する。

stream state は SoXR state、stream contract、cumulative input/output frame count、`soxr_delay` 相当の output delay を持つ。処理後に delay を更新し、output sample rate 基準の samples / ms と入力 sample rate 換算の samples を metrics に反映する。

SoXR が output を遅延させ、一部 chunk で output frame 数が 0 になることは正常な streaming 挙動として扱う。ただし SoXR process が error を返す場合、または入力も出力も進められない場合は backend process failure とする。

## diagnostics metrics

backend は次の metrics を管理し、node は diagnostics に publish する。

| metric | 算出元 |
| --- | --- |
| `algorithmic_delay_input_samples` | backend/library delay を入力 sample rate 基準へ換算 |
| `algorithmic_delay_output_samples` | backend/library delay を出力 sample rate 基準で保持 |
| `algorithmic_delay_ms` | output delay samples を output sample rate で ms 換算 |
| `processing_time_mean_ms` | process call の elapsed time total / call count |
| `processing_time_max_ms` | process call の最大 elapsed time |
| `input_frames_total` | stream state の累積 input frame count |
| `output_frames_total` | stream state の累積 output frame count |
| `expected_output_frames` | `input_frames_total * output_rate / input_rate` |
| `frame_count_error_samples` | `round(output_frames_total - expected_output_frames)` |

これらは production diagnostics に publish される backend metrics である。同じ metrics は品質比較テストでも、
処理済み backend から取得した `BackendMetrics` を GTest XML property として記録する。

## quality evidence algorithm

quality metrics は test-only evidence として計算する。production diagnostics に `rms_error`、`peak_error`、`snr_db`、`impulse_peak_offset_samples` は出さない。

test-only metrics:

- `rms_error`
- `peak_error`
- `snr_db`
- `impulse_peak_offset_samples`

評価信号:

- passband multi-tone
- out-of-band alias probe
- impulse

比較の基準:

- SoXR `VHQ` を golden reference とする。
- SoXR `MQ` / `HQ` と `internal_linear_resampler` を比較する。
- SpeexDSP quality `6` の metric test は runtime library が使える場合に実行する。runtime library が無い環境では skip される。

GTest XML には、音質差を示す test-only metrics と、backend 実行から得た delay / processing / frame count metrics を別の意味として記録する。

音質差の test-only metrics:

- `<prefix>_rms_error`
- `<prefix>_peak_error`
- `<prefix>_snr_db`
- `<prefix>_compared_samples`

backend 実行 metrics:

- `<prefix>_algorithmic_delay_input_samples`
- `<prefix>_algorithmic_delay_output_samples`
- `<prefix>_algorithmic_delay_ms`
- `<prefix>_processing_time_mean_ms`
- `<prefix>_processing_time_max_ms`
- `<prefix>_input_frames_total`
- `<prefix>_output_frames_total`
- `<prefix>_frame_count_error_samples`

## ROS graph smoke evidence

`test/cpp/test_resample_graph.cpp` は、backend を選択した状態で実際に `FaResampleNode` を起動し、
`AudioFrame` の publish / subscribe と `/diagnostics` を確認する。

検証済みの backend 選択:

- SoXR: `backend.name=soxr`、`backend.quality=MQ`
- SpeexDSP: `backend.name=speexdsp`、`backend.quality=6`

各 graph smoke は、選択された backend から `internal_linear_resampler` へ fallback していないことを
`backend.name` / `backend.quality` diagnostics で確認する。あわせて `algorithmic_delay_ms`、
`processing_time_mean_ms`、`input_frames_total`、`output_frames_total`、`expected_output_frames`、
`frame_count_error_samples` を確認する。

SoXR graph smoke は一度、高振幅 ramp 入力で `FLOAT32LE` encoding failed になった。
これは production contract を弱める理由にはしない。`fa_resample` は finite normalized range `[-1.0, 1.0]`
を要求し、graph smoke では検証目的に合う低振幅 multi-tone 入力を使う。

## known limitations

- `fa_resample` は channel 数を変えない。
- `fa_resample` は sample format / bit depth conversion を行わない。
- `fa_resample` は gain、normalize、limiter、clamp、denoise、VAD、ASR を行わない。
- `package.xml` は `libspeexdsp1` / `libsoxr0` を runtime dependency として宣言済みである。
- running container 内で `libspeexdsp1` / `libsoxr0` が見えていることは確認済みである。
- VLAbor image rebuild 後に `libspeexdsp1` / `libsoxr0` が image-persistent runtime dependency として含まれることは未検証である。
- SpeexDSP / SoXR backend を選択した状態での real device smoke と full workspace build/test は、この書類更新時点の検証済み範囲に含めない。
