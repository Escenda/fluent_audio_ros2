# fa_beamforming アルゴリズム詳細説明書

## Processing Pipeline

1. 起動時に必須 parameter を `PARAMETER_NOT_SET` として宣言する。
2. 各 parameter を必須値として読み取り、型と意味を検証する。
3. `beamforming.weights` の長さ、finite、絶対値和を検証する。
4. 入力 `AudioFrame` ごとに frame contract を検証する。
5. interleaved FLOAT32LE の各 frame から channel sample を読む。
6. `weighted_sum += sample[channel] * weights[channel]` を計算する。
7. 出力が finite かつ `[-1.0, 1.0]` の場合だけ mono FLOAT32LE として publish する。

## Buffer / Chunk

入力 chunk の frame 数は `data.size() / (expected.channels * sizeof(float))` で決まる。chunk をまたいだ状態は持たない。出力 `data` は `frame_count * sizeof(float)` だけ確保する。

## Sample Rate / Frame Size

sample rate は `expected.sample_rate` と一致する必要がある。node は sample rate conversion を行わない。frame size は入力 chunk ごとに変わってよいが、`expected.channels * sizeof(float)` の倍数でなければ drop する。

## State Machine

```text
startup
  -> declare required parameters
  -> validate config
  -> create ROS interfaces
running
  -> validate frame
  -> apply static weights
  -> publish or drop
```

config 不正時は `create ROS interfaces` に進まない。

## Backend Interface

初期実装は ROS2 node 内に static delay-and-sum style 処理を持つ。外部 DSP library や worker process は使わない。将来 backend を分ける場合も、backend は ROS2 topic 名を知らず、typed PCM buffer と weight contract だけを受け取る。

## Model Input / Output

model は使わない。入力は FLOAT32LE interleaved PCM、出力は mono FLOAT32LE interleaved PCM である。

## Threshold

数値閾値は normalized FLOAT32LE 範囲 `[-1.0, 1.0]` のみである。出力超過時に clamp や normalize は行わず、その frame 全体を drop する。

## Realtime 制約

計算は channel 数に対して線形で、動的 allocation は出力 buffer のみである。parameter 再読み込みや adaptive weight 更新は行わない。

## 使用 Library

- ROS2 `rclcpp`
- `diagnostic_msgs`
- `fa_interfaces/msg/AudioFrame`
- C++ 標準 library

## 近似・丸め・Drop 条件

FLOAT32LE の byte 列は little-endian として `float` へ復元する。weighted sum は `double` で計算し、出力時に `float` へ戻す。次の場合は drop する。

- frame contract 不一致
- data size 不正
- 入力 sample が non-finite または normalized range 外
- weighted sum が non-finite または normalized range 外

## 処理順序の理由

config 検証を最初に行うことで、weight 欠落や channel 不一致を runtime frame の fallback で隠さない。frame contract 検証を beamforming 前に行うことで、sample rate、encoding、layout の不一致を値計算で吸収しない。
