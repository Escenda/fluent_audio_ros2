# fa_packet_loss_concealment アルゴリズム詳細説明書

## 1. Pipeline

```text
AudioFrame subscribe
  -> contract validation
  -> source / epoch state check
  -> bounded PLC synthesis
  -> current frame publish
  -> baseline update
  -> diagnostics publish
```

validation は state mutation より前に行う。不正 frame が previous baseline や epoch 判定へ入ることを禁止する。

## 2. State

保持する state は直前の有効 frame だけである。

- `header`
- `source_id`
- format fields
- `data`
- `epoch`

複数 frame buffer や device buffer は持たない。`source_id` が変わった場合は、異なる source の波形を PLC に使わないため baseline を破棄し、現在 frame を新 baseline として扱う。

## 3. Gap detection

前回 frame がある場合:

```text
current.epoch <= previous.epoch
  -> duplicate / regression として drop

current.epoch == previous.epoch + 1
  -> 欠落なし。現在 frame を publish

current.epoch > previous.epoch + 1
  -> missing_frame_count = current.epoch - previous.epoch - 1
```

`missing_frame_count > plc.max_gap_frames` の場合は、音声内容を推測しない。古い baseline を破棄し、現在 frame だけを publish する。

## 4. Repeat attenuation PLC

欠落が上限以内の場合、`gap_index` を 1 から `missing_frame_count` まで進め、次の synthetic frame を作る。

```text
synthetic.epoch = previous.epoch + gap_index
synthetic.data[sample] = previous.data[sample] * attenuation_per_gap^gap_index
synthetic.stream_id = output_topic
```

これは明示的な PLC であり、隠れた fallback ではない。上限は `plc.max_gap_frames`、有効期限は gap がこの上限以内であること、ログと diagnostics counter は `concealed_frames` / `gap_resets` で観測する。

## 5. Timestamp

synthetic frame の `header.stamp` は previous frame の stamp から、previous frame の sample 数と sample rate に基づいて進める。

```text
sample_frames = previous.data.size / (channels * sizeof(float))
frame_duration_ns = sample_frames * 1e9 / sample_rate
synthetic.stamp = previous.stamp + frame_duration_ns * gap_index
```

stamp が `builtin_interfaces/Time` の範囲を超える場合は synthetic frame を出さず、gap reset として扱う。

## 6. Sample handling

扱う sample format は `FLOAT32LE / 32bit / interleaved` のみである。

- 入力 sample は finite かつ normalized `[-1.0, 1.0]` に制限する。
- attenuation は `0.0 < value <= 1.0` のため、synthetic sample は前回 sample の範囲を超えない。
- resampling、format conversion、padding、device I/O は行わない。

## 7. Realtime constraints

処理量は frame byte 数に比例する。保持 state は直前 frame 1 枚分のみであり、欠落が大きい場合も `plc.max_gap_frames` を超えて synthetic frame を作らない。
