# fa_tts 仕様書

## 目的

`fa_tts` は text-to-speech generation node である。`Speak` service request を受け、合成した `FLOAT32LE/32` `AudioFrame` を `audio/tts/frame` へ publish する。`src/processing/generation` の node として、責務は text request から audio data-plane representation を生成することに限定する。recognition / classification model node は `src/ai`、dialogue policy や LLM / MCP / VLM orchestration は `src/apps` に置く。現在の backend は `pyopenjtalk` であり、OpenAI などの cloud TTS には依存しない。

## 入力 topic

なし。

## service

| service | type | 内容 |
| --- | --- | --- |
| `speak` | `fa_interfaces/srv/Speak` | text / voice 指定を受けて音声合成する |

## 出力 topic

| parameter | default | 内容 |
| --- | --- | --- |
| `output_topic` | `audio/tts/frame` | 合成済み音声 |

## parameters

| parameter | 意味 |
| --- | --- |
| `backend.name` | TTS backend。現在は `pyopenjtalk` 固定 |
| `backend.openjtalk_dict_dir` | pyopenjtalk / OpenJTalk dictionary directory。`pyopenjtalk` backend では必須 |
| `default_voice` | request voice が空の場合に使う voice |
| `output_topic` | TTS frame publish topic |
| `cache_dir` | PCM cache directory。既定は `~/.cache/fluent_audio/tts`。空なら disk cache 無効 |

## lifecycle

起動時に `backend.name`、`backend.openjtalk_dict_dir`、pyopenjtalk runtime、parameter を検証する。`backend.name` が空または未知の場合は起動失敗する。`pyopenjtalk` backend が選択され、dictionary directory が空または存在しない場合、runtime がない場合も起動失敗する。`$HOME/.pyopenjtalk` などの暗黙 fallback は使わない。

## QoS

TTS output は reliable QoS を使う。audio frame を command state と混同しない。

## error handling

空 text、`request.play=true`、`request.volume_db != 0.0`、synthesis failure、cache write failure は response に明示する。runtime import failure や non-normalized waveform を握りつぶさない。

## backend selection

現在は `pyopenjtalk` backend のみ。backend 実装は `fa_tts_py/backends/` に分離し、ROS2 topic / message を参照しない。別 backend を追加する場合も `backend.name` を明示 config にする。neural model、external worker、cloud API を使う backend でも ROS2 topic / message を参照しない。

## non-goals

- ASR
- playback device I/O
- playback routing
- gain / normalize / limiter
- bit-depth conversion / PCM16 encode
- waveform scale guessing / clipping
- dialogue policy
- recognition / classification
- LLM / MCP / VLM orchestration
- cloud API fallback
