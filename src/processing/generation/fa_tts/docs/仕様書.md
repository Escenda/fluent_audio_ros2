# fa_tts 仕様書

## 目的

`fa_tts` は text-to-speech generation node である。`Speak` service request を受け、合成した `FLOAT32LE/32` `AudioFrame` を明示設定された topic へ publish する。`src/processing/generation` の node として、責務は text request から audio data-plane representation を生成することに限定する。recognition / classification model node は `src/ai`、dialogue policy や LLM / MCP / VLM orchestration は `src/apps` に置く。現在の backend は `pyopenjtalk` であり、OpenAI などの cloud TTS には依存しない。

## 仕様 ID

| ID | 内容 |
| --- | --- |
| `FA-TTS-SPEC-001` | `fa_tts` は `Speak` service request から `FLOAT32LE/32` `AudioFrame` を生成する text-to-speech generation node であり、playback device I/O や routing を行わない。 |
| `FA-TTS-SPEC-002` | `backend.name`、`backend.openjtalk_dict_dir`、output identity、QoS、cache 設定は起動時に明示設定され、required parameter の欠落や型不一致は起動失敗にする。 |
| `FA-TTS-SPEC-003` | `backend.name` は必須で、未知 backend は起動失敗にする。現在の実装 backend は `pyopenjtalk` のみであり、cloud API fallback を持たない。 |
| `FA-TTS-SPEC-004` | `pyopenjtalk` backend では concrete dictionary directory が必須で、`$HOME/.pyopenjtalk` などの暗黙 fallback を使わない。 |
| `FA-TTS-SPEC-005` | backend は ROS2 topic / message を知らず、text / voice から normalized `FLOAT32LE/32` payload を返す境界に閉じる。 |
| `FA-TTS-SPEC-006` | pyopenjtalk waveform が normalized `[-1.0, 1.0]` 範囲外の場合、scale 推測、clip、PCM16 encode を行わず failure とする。 |
| `FA-TTS-SPEC-007` | `request.play=true` と `request.volume_db != 0.0` は generation node の責務外として拒否し、playback routing / gain は後段 node に分離する。 |
| `FA-TTS-SPEC-008` | `request.cache_key` は 40 lowercase hex characters のみ受け付け、任意文字列や path traversal 可能な値は拒否する。 |
| `FA-TTS-SPEC-009` | cache metadata は `encoding=FLOAT32LE`、positive sample rate / channels、`bit_depth=32` を要求し、invalid cache は正常 payload として扱わない。 |
| `FA-TTS-SPEC-010` | cache write failure、synthesis failure、empty text、invalid voice は response に明示し、publish 前に失敗 response を返す。 |
| `FA-TTS-SPEC-011` | launch file は `node_name` / `config_file` を明示引数とし、package-local default config を推測しない。 |
| `FA-TTS-SPEC-012` | pytest contract は `colcon test` から実行される。 |

## 入力 topic

なし。

## service

| service | type | 内容 |
| --- | --- | --- |
| `speak` | `fa_interfaces/srv/Speak` | text / voice 指定を受けて音声合成する |

## 出力 topic

| parameter | 内容 |
| --- | --- |
| `output_topic` | 合成済み音声の publish topic |

## parameters

| parameter | 意味 |
| --- | --- |
| `backend.name` | TTS backend。現在は `pyopenjtalk` 固定 |
| `backend.openjtalk_dict_dir` | pyopenjtalk / OpenJTalk dictionary directory。`pyopenjtalk` backend では必須。`fa_tts` node には展開済みの concrete path を渡す |
| `default_voice` | request voice が空の場合に使う voice |
| `output_topic` | TTS frame publish topic |
| `output.source_id` | publish する `AudioFrame.source_id`。空は禁止 |
| `output.stream_id` | publish する `AudioFrame.stream_id`。`output_topic` / `output.source_id` と同一は禁止 |
| `cache_dir` | PCM cache directory。空なら disk cache 無効 |
| `qos.depth` | publish QoS depth。1 以上 |
| `qos.reliable` | true なら reliable、false なら best effort |

`request.cache_key` が指定された場合は 40 lowercase hex characters だけを受け付ける。任意文字列、相対 path、`..`、`/`、`\` は cache key として扱わない。

## lifecycle

起動時に `backend.name`、`backend.openjtalk_dict_dir`、output identity、QoS、pyopenjtalk runtime、parameter を検証する。required parameter が未宣言または型不一致の場合は起動失敗する。`backend.name` が空または未知の場合は起動失敗する。`pyopenjtalk` backend が選択され、dictionary directory が空または存在しない場合、runtime がない場合も起動失敗する。`$HOME/.pyopenjtalk` などの暗黙 fallback は使わない。

`fluent_audio_system` から起動する場合、system config の inline parameter で `${env:FLUENT_AUDIO_OPENJTALK_DICT_DIR}` を concrete path へ展開してから `fa_tts` へ渡せる。`fa_tts` 自身は `${env:...}` を解釈せず、受け取った path をそのまま検証する。

## QoS

TTS output QoS は `qos.depth` と `qos.reliable` で明示する。audio frame を command state と混同しない。

## error handling

空 text、`request.play=true`、`request.volume_db != 0.0`、invalid cache key、synthesis failure、cache write failure は response に明示する。cache write failure は成功扱いせず、`AudioFrame` publish 前に失敗 response を返す。runtime import failure や non-normalized waveform を握りつぶさない。

## backend selection

現在は `pyopenjtalk` backend のみ。backend 実装は `fa_tts_py/backends/` に分離し、ROS2 topic / message を参照しない。別 backend を追加する場合も `backend.name` を明示 config にする。neural model、external worker、cloud API を使う backend でも ROS2 topic / message を参照しない。

## non-goals

- ASR
- playback device I/O
- playback routing
- gain / normalize / limiter
- bit-depth conversion / PCM16 encode
- waveform scale guessing / clipping
- dialogue policy
- recognition / classification
- LLM / MCP / VLM orchestration
- cloud API fallback
