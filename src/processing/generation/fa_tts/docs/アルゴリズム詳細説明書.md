# fa_tts アルゴリズム詳細説明書

## 処理手順

```text
Speak request
  ↓
validate text / voice
  ↓
resolve cache key
  ↓
load PCM cache or synthesize with pyopenjtalk
  ↓
validate normalized waveform scale
  ↓
encode PCM16LE AudioFrame
  ↓
publish output
```

## waveform contract

pyopenjtalk backend は mono waveform が finite かつ normalized `[-1.0, 1.0]` であることを要求する。`int16` scale の推測、`32768.0` による正規化、clip は行わない。contract を満たさない waveform は synthesis failure として扱う。

## frame / buffer 前提

- 出力は backend が明示した `SynthesizedAudio.encoding` を使う。現在の `pyopenjtalk` backend は `PCM16LE/16` を返す。
- 現在は mono。
- `AudioFrame.source_id` は `fa_tts`、`stream_id` は `output_topic`、`layout` は `interleaved`。
- `AudioFrame.epoch` は playback routing の責務なので `fa_tts` では設定しない。

## rejected request fields

- `request.play=true` は playback routing の責務なので拒否する。
- `request.volume_db != 0.0` は dynamics / routing 側の gain 責務なので拒否する。

## known limitations

- backend は pyopenjtalk のみ。
- cache metadata は簡易 text file だが、必須 key 欠落や不正値を検出した場合は cache miss として再合成する。
- cache metadata には `encoding` を必須で保存する。旧 metadata は cache miss として扱う。

## backend boundary

`fa_tts_py/backends` は ROS2 topic、ROS message、`rclpy` を参照しない。ROS2 node は `Speak` service、cache、`AudioFrame` publish の adapter として振る舞う。
