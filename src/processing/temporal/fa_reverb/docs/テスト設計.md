# fa_reverb テスト設計

## 1. Scope

初期実装では unit pytest contract と backend gTest により package layout、required parameter 宣言、frame validation、DSP state 更新の fail-closed 条件を検証する。
ROS2 launch 結合と実音声 fixture は後続の integration / launch test で扱う。

## 2. 仕様 ID

| ID | 仕様 |
| --- | --- |
| `FA-REVERB-SPEC-001` | package layout が標準構成を満たす |
| `FA-REVERB-SPEC-002` | required parameter に有効な runtime default を持たない |
| `FA-REVERB-SPEC-003` | config parameter を finite / range checked する |
| `FA-REVERB-SPEC-004` | `FLOAT32LE` interleaved `AudioFrame` contract を検証する |
| `FA-REVERB-SPEC-005` | ROS topic と stream identity を別 parameter として扱う |
| `FA-REVERB-SPEC-006` | backend が input sample の finite / normalized 検証を行う |
| `FA-REVERB-SPEC-007` | backend が multi-tap feedback delay state を channel ごとに持つ |
| `FA-REVERB-SPEC-008` | output / feedback state が finite / normalized でない場合 clamp せず drop する |
| `FA-REVERB-SPEC-009` | `source_id` 変化時に backend state を reset する |
| `FA-REVERB-SPEC-010` | diagnostics に config と counters を出す |
| `FA-REVERB-SPEC-011` | colcon から pytest と backend gTest を実行できる |
| `FA-REVERB-SPEC-012` | launch の `config_file` は明示必須で package default を持たない |

## 4. Backend C++ tests

- canonical sample に deterministic feedback delay を適用する。
- channel ごとの delay line が独立していることを確認する。
- source_id change で旧 source の delayed sample が混入しないことを確認する。
- empty source / empty input / byte alignment / NaN / normalized range violation を `ProcessStatus` で返す。
- 不正 config を起動時例外として拒否する。
- 未定義の backend status message は fallback 文字列へ丸めず例外にする。

## 5. 未検証範囲
