# fa_trim アルゴリズム詳細説明書

## 1. Processing Pipeline

1. required parameters を宣言し、全項目を起動時に検証する。
2. `input_topic` から `AudioFrame` を受信する。
3. ROS node が `source_id`、`stream_id`、format、payload size を検証する。
4. ROS-free backend `internal_frame_trim` が FLOAT32 sample と trim 後 frame 数を検証する。
5. backend が `trim.leading_frames` と `trim.trailing_frames` から残す byte range を計算し、成功時だけ output payload を commit する。
6. node が `epoch` wrap を検証する。
7. `out = in` で metadata を継承し、`stream_id` と `epoch` と `data` だけを更新する。
8. `output_topic` へ publish する。

## 2. sample frame の定義

この node でいう sample frame は、全 channel の同一時刻 sample の集合である。
`expected.channels = 2` の場合、1 sample frame は `float left + float right` の 8 bytes である。

```text
bytes_per_frame = expected.channels * sizeof(float)
input_frame_count = data.size() / bytes_per_frame
output_frame_count = input_frame_count - trim.leading_frames - trim.trailing_frames
```

`data.size()` が `bytes_per_frame` の倍数でない場合、interleaved FLOAT32LE contract が壊れているため drop する。

## 3. Trim Range

残す range は次の通り。

```text
start_byte = trim.leading_frames * bytes_per_frame
byte_count = output_frame_count * bytes_per_frame
```

先頭・末尾の削除は frame 内だけで完結する。前後 frame から補完しない。

## 4. Epoch Contract

payload が変わるため、出力 `epoch` は `input.epoch + 1` とする。
`input.epoch == UINT32_MAX` の場合は wrap によって変換回数が曖昧になるため drop し、`epoch_overflow_drops` を増やす。

`header` は入力 frame の capture boundary として保持する。leading trim 後の最初の sample 時刻に変更しない。
timestamp を補正する必要がある場合は timestamp 専用 node を pipeline に明示する。

## 5. Drop 条件

| 条件 | counter |
| --- | --- |
| null frame | `contract_drops` |
| `source_id` / `stream_id` 欠落 | `contract_drops` |
| `stream_id` mismatch | `contract_drops` |
| format mismatch | `contract_drops` |
| payload size mismatch | `contract_drops` |
| non-finite / range 外 sample | `invalid_sample_drops` |
| trim 後に sample frame が残らない | `trim_exhausted_drops` |
| epoch increment wrap | `epoch_overflow_drops` |

drop した frame は publish しない。empty `AudioFrame` は出さない。

`internal_frame_trim` が返す未知の `ProcessStatus` は `std::logic_error` で fail closed する。

## 6. Realtime 制約

この node は frame 間 state を持たない。copy 量は出力 payload size に比例する。
大きな frame を処理する場合でも、暗黙の分割・retry・zero padding は行わない。

## 7. 使用 library

標準 C++17、`rclcpp`、`diagnostic_msgs`、`fa_interfaces` のみを使う。backend は標準 C++17 のみで、ROS2 依存を持たない。
外部 DSP library や external backend process は使わない。
