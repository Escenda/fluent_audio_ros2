# fa_trim テスト設計

## 1. 方針

`fa_trim` の unit test は package contract を pytest で検証し、ROS-free backend の byte列処理は gTest で検証する。
C++ node の起動・ROS2 pub/sub 結合は launch / integration test に分けるが、初期実装では `.gitkeep` のみ置く。

## 1.1. traceability

| test ID | 対応仕様 | 観点 |
| --- | --- | --- |
| `FA-TRIM-TC-001` -> `FA-TRIM-SPEC-001` | source identity | 入力 `AudioFrame` が `source_id` と `stream_id` を必須にすること。 |
| `FA-TRIM-TC-002` -> `FA-TRIM-SPEC-002` | stream identity | 入力 `stream_id` が `input_stream_id` と一致すること。 |
| `FA-TRIM-TC-003` -> `FA-TRIM-SPEC-003` | format contract | 入力 format が expected sample rate / channels / FLOAT32LE / 32 bit / interleaved と一致すること。 |
| `FA-TRIM-TC-004` -> `FA-TRIM-SPEC-004` | data alignment | `data` が non-empty で channel frame 境界に整列していること。 |
| `FA-TRIM-TC-005` -> `FA-TRIM-SPEC-005` | output metadata | output が `stream_id` を更新し、他 metadata を保持すること。 |
| `FA-TRIM-TC-006` -> `FA-TRIM-SPEC-006` | epoch policy | output epoch は `input.epoch + 1` とし、wrap を drop すること。 |
| `FA-TRIM-TC-007` -> `FA-TRIM-SPEC-007` | empty output drop | trim 後に sample frame が 0 個になる frame を publish しないこと。 |
| `FA-TRIM-TC-008` -> `FA-TRIM-SPEC-008` | sample validation | non-finite / out-of-range sample を drop すること。 |
| `FA-TRIM-TC-009` -> `FA-TRIM-SPEC-009` | topic / stream identity | topic と stream ID を分離し、衝突を fail closed すること。 |
| `FA-TRIM-TC-010` -> `FA-TRIM-SPEC-001` | backend coverage | backend が trim、channel frame、reject status、unknown status fail closed を検証すること。 |
| `FA-TRIM-TC-011` -> `FA-TRIM-SPEC-009` | package layout | 標準 layout と backend docs/source を検証すること。 |
| `FA-TRIM-TC-012` -> `FA-TRIM-SPEC-009` | colcon coverage | pytest と backend gTest が `colcon test` から実行されること。 |

## 2. 仕様とテストケース

| Test ID | Spec ID | 内容 | 実装 |

| FA-TRIM-TC-012 | build contract | colcon から pytest と backend gTest contract が実行される | `test_colcon_runs_pytest_and_backend_gtest_contracts` |
