# fa_silence_removal テスト設計

## 1. 目的

`fa_silence_removal` が silent chunk を publish せずに drop し、non-silent chunk と hangover chunk だけを元 metadata のまま publish することを確認する。

## 2. 単体 contract tests

`test/unit/test_fa_silence_removal_audio_frame_contract.py` で以下を確認する。

- `config/default.yaml` が FLOAT32LE interleaved 契約を示す
- `threshold.rms` と `hangover_ms` が明示されている
- 起動時設定が範囲検証され、不正設定で fallback しない
- `hangover_ms` が `expected.sample_rate` から sample 数へ変換される
- frame contract 検証後に RMS 計算へ進む
- invalid sample を clamp / normalize せず drop する
- silent chunk branch が publish を呼ばない
- hangover branch が chunk を publish し、残量を減らす
- publish frame は `stream_id` のみ更新し、`data` を書き換えない
- diagnostics が threshold、hangover、drop counters を出す
- colcon から pytest が実行される

## 3. C++ ビルド検証

代表コマンド:

```bash
timeout 300 colcon build --packages-select fa_silence_removal
```

確認観点:

- `fa_interfaces` に依存してビルドできる
- `diagnostic_msgs` の publisher が解決できる
- launch/config/install 対象が package share に入る

## 4. 統合テスト案

将来の integration test では、以下の fixture stream を使って ROS2 topic 経由で確認する。

| ケース | 入力 | 期待 |
| --- | --- | --- |
| non-silent | RMS が threshold 以上 | 1 message publish |
| silent | RMS が threshold 未満 | publish なし |
| active then silent in hangover | active 後、hangover 残量内の silent | silent も publish |
| active then silent after hangover | hangover 消費後の silent | publish なし |
| invalid sample | NaN / inf / out-of-range | publish なし、invalid counter 増加 |
| contract mismatch | encoding / channels / stream_id mismatch | publish なし、invalid counter 増加 |

## 5. launch test 案

`launch/fa_silence_removal.launch.py` を起動し、以下を確認する。

- `node_name` launch argument が node 名に反映される
- `config_file` launch argument で別 YAML を渡せる
- default config で node が起動する
- 不正 config では node が起動せず fatal log を出す

## 6. 未対象

この package の test では device I/O、resampling、ASR、VAD state publication、音声品質の主観評価を扱わない。
