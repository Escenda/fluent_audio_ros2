# FluentAudio テスト設計

## 1. テスト方針

FluentAudio のテストは「起動できる」だけではなく、責務境界と fail-closed 契約を検証します。

root の本資料は package 横断のテスト方針を定義します。各 package の具体 test case は各 package 配下の `docs/テスト設計.md` を正とします。

## 2. 必須テスト分類

| 分類 | 目的 |
| --- | --- |
| `test/unit` | config parser、backend factory、pure function、contract validation |
| `test/integration` | 複数 node の topic wiring と frame contract |
| `test/launch` | launch argument、params_file、system config の fail-closed |
| `test/fixtures` | 最小 frame、invalid frame、config sample、dummy model path |

## 3. Package 共通観点

各 node package は次を検証します。

- default config が仕様書と一致する
- required parameter 欠落で起動失敗する
- unknown backend は起動失敗する
- missing model / executable / endpoint は起動失敗する
- input frame format mismatch を隠れて変換しない
- backend が ROS2 message に依存しない
- runtime error を暗黙 retry / fallback で隠さない

root の `test/unit/test_repository_structure_contract.py` は repository 横断の構造契約を検証します。

- `src/**/package.xml` を持つ package は標準 docs と `test/unit` / `test/integration` / `test/launch` / `test/fixtures` を持つ
- 旧 package 名 `fa_capture` / `fa_output` を `src/` 配下に残さない
- runtime backend implementation は `rclpy` / `rclcpp` / `fa_interfaces` を import / include しない

## 4. System Config 観点

`fluent_audio_system` は次を検証します。

- missing config path は起動失敗
- missing `system` / `groups` は起動失敗
- missing `params_file` は起動失敗
- missing `params_file` を inline parameter だけで補わない
- disabled group / node は展開しない
- `${share:package}` の解決失敗は起動失敗
- remapping schema が不正なら起動失敗

## 5. VLAbor 連携観点

VLAbor profile 連携では次を検証します。

- `type: include` で `fluent_audio_system/run.py` を呼べる
- `fluent_audio_enabled=false` では起動 action が無効
- profile には enable、config path、source/sink binding だけを置く
- backend / model 詳細は FluentAudio system config と node config に閉じる
- parent repo の gitlink update を commit しない運用を守る

## 6. 未検証の扱い

実 device、GPU、cloud API、external worker を必要とする経路は unit test の成功だけで完了扱いにしません。代替 smoke test を行った場合は、最終報告で本番相当経路が未検証であることを明記します。
