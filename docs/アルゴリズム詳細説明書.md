# FluentAudio アルゴリズム詳細説明書

## 1. 全体 pipeline

FluentAudio の標準 pipeline は次の層で構成します。

```text
source backend
  -> fa_in
  -> format / correction / dynamics / frequency / temporal / spatial nodes
  -> streaming nodes
  -> non-AI analysis / ai / generation / routing nodes
  -> fa_out
  -> sink backend
```

`fa_in` と `fa_out` は source / sink adapter であり、音声編集を実行しません。音声変換や解析は必ず dedicated processing node に分離します。

## 2. 処理分類

`src/processing` の直下は次の 9 分類で固定します。AI model node とリアルタイム伝送 node は責務が大きく異なるため、top-level の `src/ai` と `src/streaming` に分離します。

| category | 内容 |
| --- | --- |
| `format` | resample、bit depth、channel、interleave、codec、value range |
| `dynamics` | gain、normalize、compressor、limiter、expander、gate、AGC |
| `frequency` | EQ、low-pass、high-pass、band-pass、notch、de-esser、spectral subtraction、Wiener |
| `temporal` | trim、silence removal、time stretch、pitch shift、delay、reverb、fade、windowing |
| `correction` | denoise、AEC、dereverb、declip、declick、hum removal、DC offset |
| `spatial` | pan、downmix、upmix、beamforming、source separation |
| `analysis` | STFT、mel、MFCC、loudness measurement などの非 AI feature extraction |
| `generation` | TTS、voice conversion、speech enhancement、speech separation、neural codec、vocoder などの audio data-plane generation / transformation |
| `routing` | mixer、bus routing、sidechain、ducking、monitor、loopback、patchbay |

Top-level の追加分類:

| directory | 内容 |
| --- | --- |
| `src/ai` | VAD、KWS、ASR、Turn Detector、speaker embedding、audio embedding、SED などの AI model node |
| `src/streaming` | buffering、jitter buffer、clock drift、packet loss concealment、latency compensation、time alignment、chunk overlap、overlap-add |

`src/processing/analysis` は AI package の置き場ではありません。VAD、KWS、ASR、Turn Detector、SED、speaker / audio embedding のように model inference または認識結果生成が主責務の node は `src/ai` に置きます。streaming node も processing の下には置かず、top-level の `src/streaming` に置きます。

`src/processing/generation` は「音声を別の音声表現へ作り替える」data-plane 処理だけを扱います。TTS や neural vocoder が内部で model / external worker / cloud API を使う場合でも、認識、分類、会話制御、LLM / MCP / VLM orchestration、dialogue policy は generation ではありません。これらは AI node なら `src/ai`、system behavior を束ねる app なら `src/apps` に置きます。

## 3. Node / Backend 分離

ROS2 node の処理手順:

1. parameter を読み、必須値を検証する。
2. backend config を明示型へ変換する。
3. backend capability と node が受け付ける `AudioFrame` contract を確定する。
4. backend を `backend.name` で構築し、model / provider / executable / endpoint / payload encoding が capability と一致することを検証する。
5. input frame の source identity、stream identity、sample rate、channel、encoding、bit depth、layout、byte alignment、value range を検証する。
6. backend に ROS2 非依存の byte / float buffer と metadata を渡す。
7. backend result の schema、値域、metadata を検証する。
8. backend result を ROS2 message に変換して publish する。

Backend は ROS2 topic、namespace、launch、parameter server を参照しません。

## 4. ASR Backend Boundary

現行標準 ASR backend は `parakeet_multilingual_buffered` です。
この backend は multilingual Parakeet 1.1B `.nemo` を `fa_asr` backend/session 内部の runner として保持し、16 kHz mono `FLOAT32LE` の ASR-ready stream だけを受け付けます。

`parakeet_multilingual_buffered` の streaming policy は full-context model を前提にした rolling buffer / chunked re-decode です。
partial は未確定 hypothesis であり、VAD / TurnDetector / timeout close で `finish()` が返す final だけを committed transcript として扱います。
speech energy が十分ある audio に対して final transcript が空なら成功扱いしません。

language を profile が保証できない場合、`backend.language` は空文字、`backend.language_policy` は `auto_detect` とします。
固定 language を暗黙に補完しません。

unsupported model、Whisper、OpenAI、NIM、Riva、gRPC、stdin/stdout JSONL worker は標準経路、default、参照実装、validation path ではありません。
これらを使う場合は optional / legacy backend として明示選択し、`parakeet_multilingual_buffered` の失敗時 fallback にしません。

Python version、venv、GPU runtime、native library が ROS2 node と一致しない engine は external worker / process / container 境界に置けます。
ただし external backend は command、model path、endpoint、credential contract を起動時に検証し、条件を満たせない場合は起動失敗にします。
external backend の成功・失敗は現行標準 backend の成功・失敗と混ぜません。

## 5. Format Handling

Hidden format negotiation は禁止します。analysis backend が必要とする canonical format は node config に明示し、入力 frame が一致しない場合は reject / fail closed します。

Resample、downmix、bit depth conversion、normalized range conversion が必要な場合は `src/processing/format` の node を pipeline に挿入します。

未対応 format は「変換できなかった入力」ではなく「この backend の capability 外の入力」です。例えば `FLOAT32LE/32/interleaved/mono/16kHz` を要求する AI backend に `PCM16LE`、`PCM32LE`、stereo、planar、sample-rate mismatch、non-finite sample が入った場合は、変換や clamp を行わず frame rejection とします。対応形式を増やす場合は、その形式を backend capability として仕様化し、対応する validation と behavioral test を追加します。
